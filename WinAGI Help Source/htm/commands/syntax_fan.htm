AGI Logic FAN Syntax

When the first set of fan created AGI tools were written, the exact syntax that was used by Sierra to program AGI games was unknown. Some small samples of source code were available, which has offered a glimpse of what it was probably like, but no official documentation was known of at the time that described Sierra's syntax in detail.

In those early days, a set of syntax rules was agreed on by those involved in making new AGI tools. Those rules are presented here. WinAGI is fully compatible with these rules.

In later years, the complete source code for a handful of games has become available, and Sierra's logic compiler CG.EXE has been fully disassembled. The original Sierra syntax is now fully known. WinAGI includes the option to design AGI games using the original official Sierra syntax. See the AGI Logic SIERRA Syntax topic for detailed information on Sierra syntax rules.

Syntax Overview

AGI Fan syntax is based loosely on the C family of languages; it uses a small set of keywords to manage program flow, and predefined commands that provide access to AGI's underlying set of script actions. Source code is converted into the pseudo-code which AGI uses to manage the game's graphics, text and sound resources as well as all player input.

All AGI command names are pre-defined, unlike Sierra's original syntax where every command had to be defined in each logic in order for the compiler to read the source code.

The types of arguments used by commands are also pre-defined. Arguments for most commands are limited to byte value range (0 - 255). The said command, which tests the parsed text entered by the player is the one exception, which uses 16 bit unsigned numbers.

To improve readability of source code, game programmers can use the #define keyword to substitute meaningful names for argument values.

All tokens (commands, keywords, define names, argument markers) are case sensitive. So 'Addn' is not the same as 'addn' for example.

Action Commands

Action commands represent the user program functions that are available in AGI. Each action command is converted to the appropriate byte value that gets stored in the logic's compiled pseudo-code. Action commands are specified by the command name followed by parentheses which contain the arguments, separated by commas. A semicolon is used to mark the end of a command. The parentheses are required even if there are no arguments. Action commands have no return value.

The format for arguments is "x##' where 'x' is the prefix for the type of argument, and ## is a valid number value for the argument. This ensures the programmer does not inadvertently use the wrong argument type in a command.

assign.v(v50, 0);
program.control();

Multiple commands may be placed on the same line:

reset(f6); reset(f7);

Alternate Action Syntax

Substitutions for the following action commands are available:

increment(v30);       v30++; or ++v30;
decrement(v30);       v30--; or --v30;  
assignn(v30,4);       v30 = 4;
assignv(v30,v32);     v30 = v32;
addn(v30,4);          v30 = v30 + 4; or v30 += 4;
addv(v30,v32);        v30 = v30 + v32; or v30 += v32;
subn(v30,4);          v30 = v30 - 4; or v30 -= 4;
subv(v30,v32);        v30 = v30 - v32; or v30 -= v32;
mul.n(v30,4);         v30 = v30 * 4; or v30 *= 4;
mul.v(v30,v32);       v30 = v30 * v32; or v30 *= v32;
div.n(v30,4);         v30 = v30 / 4; or v30 /= 4;
div.v(v30,v32);       v30 = v30 / v32; or v30 /= v32;
lindirectn(v30,4);    *v30 = 4;
lindirectv(v30,v32);  *v30 = v32;
rindirect(v30,v32);   v30 = *v32;
set.string(s5,m20);   s5 = m20; or s5 = "msg text";
set(f99);             f99 = true;
reset(f99);           f99 = false;

When using the alternate forms for indirection (lindirectn, lindirectv and rindirect), the asterisk character is prepended to the variable argument. No space is allowed between the asterisk and the variable argument. This substitution can be used with define names as well as with variable argument markers:

#define avar  v99
*avar = 1; [ same as '*v99 = 1;' or 'lindirectn(avar, 1);'

Pointers

WinAGI includes the ability to pass non-numeric arguments as 'pointers' using the '&' character. For example, '&v99' will compile as '99', '&s4' will compile as '4', etc. You can also use the pointer character with define names, for example:

#define tempVar  v101
v99 = &tempVar; [ same as v99 = 101;

Pointers are useful when working with indirect arithmetic to make code easier to read and modify. For example:

#define array0     v100 [ change define values to modify the array 
#define arrayEnd   v110 [ without having to rewrite the code block
#define work       v255
[
[ loop to clear an array
work = &array0;   [ start at beginning of array
:loop1
  *work = 0;      [ zero out this element
  ++work;         [ move to next element
if (work <= &arrayEnd)
  {
  [ if not past the end, do the next element
  goto(loop1);
  }

While less useful, the pointer syntax also works for any argument type, for example '&c1', '&f1', '&i1', '&m1' '&o1', '&s1' and '&w1' will all compile as '1'.

Pointer syntax is a WinAGI-specific feature, and is not supported by AGI Studio.

The return command

The return command is a normal action command (number 0), with no arguments. The return command can be used anywhere in the logic, but the last command of the logic MUST be a return command.

Test Commands

Test commands return a value of true or false are only used with the if keyword. As with action commands, arguments for the command must be enclosed in parentheses, separated by a comma, and the parentheses are required even if there are no arguments for the command.

if (isset(f5)) ......
if (greatern(v5, 6)) ......
if (have.key()) ......

An exclamation point (!) placed in front of a command signifies a NOT, which toggles the result of the test command:

if (!isset(f7)) ......
if (!greatern(v5, 7)) ......

Alternate Test Syntax

Substitutions for the following test commands are available:

equaln(v30, 4)        [ v30 == 4
equalv(v30, v32)      [ v30 == v32
greatern(v30, 4)      [ v30 > 4
greaterv(v30, v32)    [ v30 > v32
lessn(v30, 4)         [ v30 < 4
lessv(v30, v32)       [ v30 < v32
!equaln(v30, 4)       [ v30 != 4
!equalv(v30, v32)     [ v30 != v32
!greatern(v30, 4)     [ v30 <= 4
!greaterv(v30, v32)   [ v30 <= v32
!lessn(v30, 4)        [ v30 >= 4
!lessv(v30, v32)      [ v30 >= v32

Flags can be tested by just using the name of the flag, as an alternative to the isset command:

if (f6) .....  [  same as if (isset(f6))

Variables can be tested as either TRUE (> 0) or FALSE(== 0) by just using the name of the variable:

if (v0) .....  [  same as 'if (greatern(v0, 0))' or 'if (v0 > 0)'
if (!v0) ....  [  same as 'if (!greatern(v0, 0))' or 'if (v0 <= 0)'

The said Command

The said test command is the only AGI command that uses unsigned 16 bit (0 - 65535) arguments instead of 8 bit (0 - 255) arguments. Also, the number of arguments in a said command can vary (but must be at least 1). The arguments corresponds to the words' group numbers in the WORDS.TOK file:

if (said(4, 80)) .....

The text of the word can also be used in place of the numbers. You can use any word in the word group, although the normal convention is to use the first word in the group. Text arguments in the said command must be surrounded by quotation marks:

if (said("look")) .....
if (said("open", "door")) .....

Argument Types

There are 10 different types of arguments that commands use. Argument values must use the following prefixes:

 Type Syntax ID Prefix
 Byte byt {no prefix}
 16 Bit Number n/a {no prefix}
 Controller ctl c
 Flag flg f
 Inventory Item itm i
 Message msg m
 Screen Object obj o
 String str s
 Variable var v
 Word wrd w

Argument markers for non-numeric types consist of the prefix and then a number from 0 - 255, e.g. v5, f6, m27, o2. The programmer must use the correct argument markers in commands that match the command's required argument type.

move.obj(o4, 80, 120, 2, f66);   [ OK
move.obj(4, v80, m120, f2, c66); [ NOT OK wrong argument types

Text Arguments

Text arguments, such as those for messages and inventory items may be coded with the appropriate argument marker (m1, i2), or as literal text string. Quote marks must be used around text arguments when using literal text strings.

print("He's not here.");
print(m12);
if(has("Jetpack")) .....
if(has(i9)) .....

Literal text strings for messages can also be split over multiple lines by separating two or more literal strings with a new-line character. No other commands, keywords, symbols are allowed between the strings:

print("This message is split "
      "over multiple lines.");

[ equivalent statement:
print("This message is split over multiple lines.");

Compiler Escape Character and Special Characters

To include certain special characters such as a line return or a double-quote, WinAGI uses the backslash character '\' as an 'escape' character. The escape codes that WinAGI uses are:

    • \n - Inserts a 'new-line' character (ascii code '0x0A')
    • \" - Inserts a double quote character (ascii code '0x22'); needed because the compiler normally treats a double quote character as a string delimiter
    • \\ - Inserts a backslash character (ascii code '0x5C');

If the character following the escape character '\' is not one of the above, the escape character '\' is ignored, but the following character is kept.

Because AGI also treats the backslash as a runtime escape character, there are several situations that need special sequences to display certain characters:

    • \\\\ - Inserts a single backslash when printing or displaying text; when the string is compiled, the four backslashes become two backslashes in the actual message text used by AGI, and then when AGI draws the message text on the screen it interprets the first backslash as the runtime escape character, and then displays only the second backslash.
    • \\% - Inserts a percent sign (ascii code '\x25') when printing or displaying text; normally when AGI encounters a percent sign, it treats it as a special format code. To display a percent sign, it must be preceded by a runtime escape code (requiring "\\" in the source code)
    • "\\ " - Inserts a non-breaking space. When AGI formats a message to display when using print commands, it automatically inserts line breaks between words so text does not extend past the print area margins. It will not insert a line break between words separated with a non-breaking space.

Within a message text argument, only standard ASCII characters (1-255) may be used. Characters above 127 and less than 32 are not fully supported by AGI, but can be used if the programmer understands the restrictions and limitations. Most AGI development tools do not provide a means to add these characters, although WinAGI does.

WinAGI includes a special compiler escape code, '\x' that can be used to insert any 8-bit character into a string. The syntax is \x##, where '##' is a two digit hexadecimal value (01-FF). WinAGI will convert the code into the appropriate 8-bit character and add it to the string.

Compiler Keywords

Certain words are reserved as keywords by the compiler, and cannot be used for any other purpose. The following words are reserved keywords:

    • if
    • else
    • goto
    • #define
    • #include
    • #message

Conditional Statements (if and if ... else blocks)

An if block will execute one or more commands only if the condition of the if statement is TRUE. The test commands must be enclosed in parentheses '()', and the action commands to be executed must be enclosed in curly brackets '{}':

if (<test commands>)
    {
    <action commands>
    }

Multiple tests within the same if statement must be separated by && (representing 'AND') or || (representing 'OR'). When test commands are to be ORred together, they must be placed in parentheses. If test commands are to be ANDed together, they must not be placed in parentheses:

if((<test1> || <test2>)) [  ORed tests in parentheses
  ......

if(<test1> && <test2>)  [  ANDed tests not in parentheses
  ......

if(<test1> && (<test2> || <test3>)) [ combination of ANDed and ORed tests 
  ......

if((<test1> && <test2>) || <test2>)   [ NOT OK; ORed tests are not in parentheses

An if ... else block is similar to the if block, but will execute one block of action commands if the if statement is TRUE, and a different block of action commands if the if statement is FALSE:

if (<test commands>)
  {
  <action commands if test is TRUE>
  }
else
  {
  <action commands if test is FALSE>
  }

goto Statement

The goto keyword takes a single parameter, the name of a label. The label must be enclosed in parentheses and the command must include an ending ';' character similar to action commands:

goto(Label1);

Labels are tokens that can be used by the goto command to control program flow. The label name must conform to the rules for define names. The label must either end or begin with a single colon character:

Label1:

This is also a valid label:

:Label2

Defines

To improve the readability of source code, all arguments can be represented by aliases, known as 'defines'.

The keyword #define is used to assign argument values to a define name. The first parameter is the define name and the second parameter is the argument value that the name will substitute.

The define value must be a valid number, argument marker, or literal string.

The define name can contain letters, numbers, the underscore character (_) and any of these characters: #$%.@. No spaces are allowed. Define names must conform to the following rules:

    1. Name can't be blank
    2. Name can't be numeric
    3. Name can't be an AGI action command
    4. Name can't be an AGI test command
    5. Name can't be a compiler keyword (if, else, goto, #include, #define, #message)
    6. Name can't be an argument marker (v1, f2, o3, etc)
    7. Name can't include include space ' ' or special characters !"&'()*+,-/:;<=>?[\]^`{|}~
    8. Name can't include control codes (char value <=32) or extended characters (char value >=128)
    9. Name must be unique in its context (i.e. no duplicate names in the same local define list)
#define ego o0
#define room_descr "This is a large hall with tall pillars down each side."

Define names can be used in place of the argument value they replace:

draw(ego);
print(room_descr);

Define names can only be used as arguments of commands (including the goto command).

Including files

You can include (embed) another file in your logic source code by using the #include keyword. The filename of the file to be included follows the command, and must be enclosed in quotation marks.The filename can include path information, including relative path notation. If the file does not include path information, it is assumed to be in the same directory as the source code file:

#include "file1.txt"
#include "c:\documents\file2.txt"
#include "..\file3.txt"

When the compiler encounters an include statement, the line with the include statement is replaced with the contents of the included file before compiling begins.

Auto Including Reserved Names, Resource IDs and Global Defines

WinAGI includes options to automatically add include files to all logics that contain define statements for AGI reserved arguments and constants, resource IDs and global defines. Each of these features can be toggled on or off as desired depending on how you want to manage defines in your game. See the WinAGI AutoInclude properties topic for more information.

Scope and Hierarchy of Defines

Define names are limited in scope based on where they are located. Reserved names, resource IDs and global defines are available in all logics. Local defines are only available in the logic where they are defined.

When the compiler encounters an argument value that is not a number, argument marker or literal string (i.e. '99', v1', 'f5', "look" etc.) it will look for an define in the following order:

    1. Locally defined names
    2. Globally defined names
    3. Resource IDs
    4. Reserved names (a special group of defines for reserved values that have preset purposes in AGI)

Global Defines,

To manage define names that you wish to be global in scope use global defines. WinAGI will automatically add an #include file statement to add the 'globals.txt' file to all logics in your game.

Messages

When using commands that take messages as arguments, the compiler will automatically handle numbering the messages, and inserting the correct numerical value in the compiled code.

Messages can be explicitly assigned to a number, if desired, by using the #message keyword. The parameters of #message are the message number (1 - 255) and the text of the message:

#message 4 "You can't do that now."

Then you can use the message number as the parameter in commands:

print(m4);

Or embed the message in commands as normal and the number you assigned to it will be used:

print("You can't do that now.");  [ same as 'print(m4);'

Messages can be declared anywhere in the source code - you do not have to declare a message before you use it. The generally accepted format is to group all message declarations at the very end of the logic. Also, note that there is no message 0; message numbers start at 1. When assigning numbers, you are not required to assign them sequentially. WinAGI will automatically assign a null string to any messages that skipped over and not explicitly assigned.

Separators, White Space and Line Breaks

Commands, keywords, define names, and symbols can be separated by using white space (' ' characters and line breaks, but space is not required.) 

if(v1==1){v1+=1;}else{v1+=2;}
[ is the same as
if ( v1 == 1)
  {
  v1 += 1;
  }
else
  {
  v1 += 2;
}

Space is not allowed between a colon and a label name:

:Label1   [ OK
: Label2  [ NOT OK

Label3:   [ OK
Label2 :  [ NOT OK

All action commands, including alternate syntax must have a terminating ';' character. Goto statements also require a terminating ';' character.

Keyword statements never use a terminating ';' character.

addn(v99, 1)       [ NOT OK
addn(v100, 1);          [ OK

#define name1    f100;  [ NOT OK
#message 1 "a message"  [ OK
if (v101 == 1) { [ OK ...... } else; { [ NOT OK ...... }

Comments

In-line comments are set off by a single open square bracket ([). Although the bracket is the preferred comment marker, you can also use a double slash (//):

addn(v99, 1); [ Rest of line is ignored
addn(v99, 2); // Rest of this line is also ignored

WinAGI does not support block comments (/* ... */).
