Sierra's OBJECT File Compiler

The original tool that Sierra used to create the OBJECT file in AGI games was a simple MSDOS program called MKOBJ.EXE. The most recent publicly released version of MKOBJ.EXE is 2.10, with a copyright date of 1986.

Usage

MKOBJ.EXE is a native MSDOS program. Its usage is:

mkobj

MKOBJ.EXE has no optional settings or arguments. It requires a text file named "object.txt" to be in the same folder, and that text file must contain inventory item name and room data.

MKOBJ.EXE uses a very primitive parser; double quote marks are used to delimit text strings; numbers can only be positive integers; spaces, tabs and line breaks are used to separate tokens. Commas or semicolons can optionally be used to separate parameters.

When an error is encountered, processing stops immediately, and no output file is created.

object.txt Source File Format

The source file, "object.txt", must be a simple text file. Data entries consist of a single line of text that begin with one of two keywords .numani and .obj. Each entry must be on a separate line. Comments can be added by using the '[' character.

[
[       TYPICAL OBJECT SOURCE FILE
[

[************* ANIMATED OBJECTS
.numani 16

[************* INVENTORY OBJECTS
.obj    1,  "item1 name", 34    [ comment
.obj    2,  "item2 name", 0

The .numani token is used to set the maximum screen object value for the game. Almost all Sierra games used a value of 16, but any byte value will work. To conserve memory, the value should be as low as possible based on the number of animated objects that the game uses.

The .obj token is used to add inventory items to the OBJECT file. The syntax to use is:

.obj object#, "item name", startroom#

Commas are not required to separate the parameters, but if used, must be placed immediately after the number or name parameter without any spaces. Any extraneous commas will cause an error. The item name is not required to be enclosed in quotes, unless the item name contains spaces, commas, semicolons new line characters or quote marks. To include a quote mark in the item name, use the '\' character:

.obj 1, "item \"in-quote\" name", 0

Keywords and their parameters must all be on one line. If there are line breaks between parameters MKOBJ.EXE will raise an error and immediately quit.

Bugs and Anomalies

MKOBJ.EXE always adds the null item ("?") as the first item in the item list, whether it is actually used or not. Any item number that was not explicitly assigned a name and starting room will be assigned to the null item name with a starting room value of zero.

The .numani command is not required; if it is not included, MKOBJ.EXE will use a value of zero for the maximum screen object value.

MKOBJ.EXE treats comma characters and semi-colon characters exactly the same.

.obj 1; "itemname"; 0 [ OK

If the number parameter of the .numani command or the start room parameter of the .obj command is greater than 255, the number will be converted to a byte value. Negative numbers will raise an error.

The item number parameter of the .obj command is NOT converted to a byte value; the programmer must make sure it is not greater than 255. If the value is greater than 255, MKOBJ.EXE will create an item name table of the same value (i.e. if item number parameter is 300, there will be 300 item entries in the output file). Any items with a number greater than 255 can't be accessed by AGI, and will just waste memory when the game is run.

If more than one item is added with the same number, only the item name of the last entry will be assigned to that number; the other item name strings will still be added to the list, but will just waste space as they won't be referenced by any item number.

MKOBJ.EXE converts all cases of "\x0d\x0a" to just "\x0a". In other words, all carriage-return-linefeed pairs are replaced with just a linefeed. However, if the '\x0d' character is not followed by a '\x0a' character, it is not removed, and will cause MKOBJ.EXE to raise an unrecognized token error.

The '\x1a' character is treated as an 'end of file' token. When MKOBJ.EXE loads data from the source file, it will stop reading data immediately after the '\x1a' character; no more data from the file will be processed.

The text parser in MKOBJ.EXE is very simplistic. It handles separator characters (',' and ';') and double quotes ('"') in a non-standard manner that leads to some unusual behaviors:

    • When getting the next token from the input text, MKOBJ.EXE first skips all space and tab characters. Then it add characters to the token until it reaches a tab, space, semicolon, comma or newline '\n', and then returns the token. This means a ';' or ',' that follows white space will return a null length token. If a number was expected, the null token will raise an error. If an item name was expected, MKOBJ.EXE will use the null item text. Some examples:
      • .obj 1, "itemname1"  , 0 [ NOT OK

      Because of the space between "itemname1" and the comma, MKOBJ.EXE will read the comma as a null token when it expects the start room parameter. This will cause an error.

      .obj 2,
"itemname2", 0 [ NOT OK

      The line break between the first comma and "itemname2" will be interpreted as the item name, and then the "itemname2" token will raise the "not a number" error.

      .obj 3   , "itemname3, 0 [ NOT OK

      In this example, the comma between '3' and "itemname3" is read as a null token, which MKOBJ.EXE will treat as the "?" item; then it reads the "itemname3" token as the start room parameter which causes a 'not a number' error.

      .obj 4 ,  0 [ OK

      This example is OK; the comma is read as a null item. This is the same as

      .obj 4 "?",  0 [ OK
    • Double quotes ('"') are used to delimit strings of text for item names. They are only required if the item name needs to include a space, double quote or newline character. However, the starting and ending quotes are never included in the returned token. The read function automatically skips them before returning the token to the parser. Also, an ending double quote does NOT end a token; the read function continues to add characters until a space, tab, newline, ',' or ';' character is found.
      • abc returns abc
"abc" returns abc
"ab"cd returns abcd
"ab"c, returns abc
"123" returns 123

      This allows entries such as this:

      ".obj" "3" itemname,  0 [ OK

Item names can include the newline ('\x0a') character.

.obj 3 "line1\nline2",  0 [ OK

Although allowed, you should not include newline characters in item names. The extra line breaks interfere with the placing of text when the inventory item screen is displayed.

The read function in MKOBJ.EXE has a bug in how it handles comments. When the '[' character is encountered, the read function skips to the end of the line, but it does not increment the line counter, which is used when displaying errors. The displayed line of an error will be off by the number of comment lines that are between the start of the file and the actual line of the error.
