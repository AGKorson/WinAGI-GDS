Sierra's WORDS.TOK File Compiler

The original tool that Sierra used to create the WORDS.TOK file in AGI games was a simple MSDOS program called WORDS.EXE. The most recent publicly released version of WORDS.EXE is 3.02, with a copyright date of 1984.

Usage

WORD.EXE is a native MSDOS program. Its usage is:

word

WORD.EXE has no optional settings or arguments. It requires a text file named "words.txt" to be in the same folder, and that text file must contain the word list to be converted to the WORDS.TOK format.

WORD.EXE uses a very primitive parser; numbers can only be positive integers; spaces, tabs and line breaks are used to separate parameters.

When an error is encountered, WORDS.EXE will ignore the entry with the error and attempt to continue.

words.txt Source File Format

The source file, "words.txt", must be a simple text file. Data entries consist of statements using one of three keywords: .d, .i and .s. Comments can be added by using the '[' character.

[
[       TYPICAL WORDS.TOK SOURCE FILE
[

.d  rol         9999

.d  look           2
.s  show        look
.s  see         look
.s  read        look
.s  examine     look

.i  an          [ articles
.i  the
.i  mine        [ pronouns
.i  my
.i  we

The 'define' keyword .d is used to add a new word group. The first parameter is the word text, the second parameter is the word group number. The only separator allowed between parameter is white space (spaces, tabs, new lines). Unlike Sierra's logic compiler and object compiler, the comma and semicolon are NOT used as separators.

Words are NOT enclosed in quotes; using quotes around words will cause an error. The '$' character is used to represent a space character in word text:

.d look$in  3  [ represents 'look in'

Both the word text and the word group number must not already be in use or an error will be raised.

Group 1 is pre-defined by the compiler and the word "anyword" is assigned to it. You cannot use the .d keyword to add a group 1.

To allow the players flexibility when entering text into AGI's text parser, use the .s keyword to add synonyms for words which have already been assigned a group number. The synonym will be assigned to the same group as the existing word. The first parameter is the word text of the synonym, and the second parameter is the word text of the existing word. The associated word must already exist in the list or an error will be raised.

.s examine   look  [ 'examine' will use same group number as 'look'
.s show   examine   [ OK to use a synonym to add a synonym

The 'ignore' keyword .i is used to add words to the 'ignored' group (group number zero). The lone parameter is the word text of the word being added. The word text must not already exist in the word list or an error will be raised.

.i     the

Using the .i keyword will add a group number 0 if it has not already been added. You can add an ignored word to group zero using the .d keyword, but only if the .i keyword has not yet been used. You can also use the .s keyword to add ignored words.

.d   an   0   [ same as '.i   an'
.i   the   [ same as '.s   the   an'

The words 'a' and 'i' are not automatically added, but they don't need to be included in the words.txt source file because at run time AGI automatically assigns them a group number of zero even if they are not included in WORDS.TOK.

Using the .i keyword is equivalent to using the .s keyword with a word from of zero.

All tokens, including keywords and word text tokens, are case-insensitive. All text characters are converted to lower case before the token is processed by the compiler. This ensures there are no upper case characters in the words added to WORDS.TOK file.

.D LOOK  2  [ same as '.d look  2'

Each define, ignore and synonym statement can be split over multiple lines, but you cannot put multiple statements on the same line. WORDS.TXT ignores all characters on a line after it finishes processing a command.

.d aword 123       .d bword 124 [ 'bword' define will never get processed

Bugs and Anomalies

Group numbers are converted to a 16 bit unsigned integer value (0 - 65355). However the internal buffer for managing which groups are in use only has room for group numbers 0 - 9999. Using a group number outside this value will not throw an error, but it will result in a corrupted output file because of a buffer overflow. The value 9999 is arbitrary and not a limit of AGI itself; if properly constructed, a WORDS.TOK file can actually use any valid unsigned integer value as a word group and AGI will handle it without error. (WinAGI allows use of group numbers greater than 9999.)

There is a bug in the comment handler in WORDS.EXE. When a '[' character is encountered, WORDS.EXE skips to the end of the line, increments the line counter, then continues checking characters beginning on the next line. This means you need to make sure to always precede the '[' character with white space, or you may get unexpected results.

.d aword  1234[ badcomment
.d bword  1235

[ WORDS.EXE reads this as:
.d aword 1234.d
bword 1235

Another bug related to comments is that the comment marker '[' is actually not required at the end of a define, ignore, or synonym statement. Once WORDS.EXE successfully finishes processing the statement, it skips all characters until another new line '\n' is found. However, if an error occurs, the rest of the line is NOT skipped.

.d aword 123       .d bword 124 [ 'bword' define will never get processed
.d aword xxx       .d bword 124 [ 'aword' raises error, 'bword' gets added

If the last line of the source file 'words.txt' is a comment, it should include a line break ("\r\n") at the end. If the line break is not included, WORDS.EXE will raise an error (but the file is still created sucessfully.)

Do NOT use a word that is longer than 39 characters. WORDS.EXE will cause a buffer overflow when trying to read the word which will freeze and crash the program.

WORDS.EXE automatically adds the word "anyword" to group 1 when it starts up.

The newline character ('\x0a' or '\n') does not act as a token separator; it just increments the line count to manage error messages. However, the carriage return character ('\x0d' or '\r') does act as a token separator. Since it was common practice in the days of Sierra to use the string "\r\n" as a line break, this effect is not noticeable. If you create the source file 'words.txt' in an editor that uses only '\n' as a line break you will get unexpected results.

WORDS.EXE will raise an error if you try to start a word with anything other than a letter a-z. However, it will add any and all characters after that, even if they are characters that AGI will not process at runtime. The programmer is responsible to make sure no unallowed characters are included in word text.

.d run-run!  2  [ will compile, but will never match in a said command

When reading numbers for the .d group number parameter, WORDS.EXE returns all characters until a separator is found, then converts all valid digits at the start of the token into a number value.

.d look 2abc  [ same as '.d look 2'
