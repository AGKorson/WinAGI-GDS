Sierra's Logic Script Compiler

Sierra AGI game programmers used a tool named CG.EXE to convert their text based script source into logic byte code. It appears to have been in use since the early days of AGI, probably being written around the time of AGI version 1. The key feature of CG.EXE is that it did not have AGI's command syntax built in- it required each script to include command definitions, which CG.EXE would use to generate the desired byte code.

Only one version of CG.EXE, v3.14, has been made public. This reference is specific to that version.

Using CG.EXE

The file creation date of CG.EXE is 1984, which coincides with early version 2 of AGI. It does not appear that it was ever updated, even as AGI progressed through version 3. Most likely this is because there was no difference in logic resource structure, the additional commands added in later versions was easily handled by CG.EXE.

CG.EXE is a native MSDOS program. The usage form is:

cg room [room...] [-o output_directory] [-b buffer_size] [-v]

There are three possible command switches:

    • -o: Sets the output directory (where the compiled logic files will be saved). The default is the current directory. The output directory can be any legal MSDOS directory name, including relative paths. If the argument is not valid directory name, CG.EXE will quit with an error. If the argument is a valid directory name, but does not exist, CG.EXE will run, but will raise an error at the end when it can't write the output files.
    • -b: Adjusts the file write buffer size - The default is 1000h (4K). There's really no reason to change this, but it can be adjusted if desired. If the argument following the -b switch is non-numeric or missing, a value of zero is used, which will cause CG.EXE to fail immediately. Larger values means disk writes are minimized, but take up more memory.
    • -v: Sets 'verbose' mode. This switch does not take an argument. When used, it causes the output to display additional information, including the number of symbols and messages found in the source file, and information on the symbol hash table.

Arguments (command switches and source files) can be in any order. Each argument, including source code files ('room') must be separated by one or more spaces. Filenames have to be valid MSDOS filenames (no long-filenames) and can include path info and wildcard characters ('?' and '*').

If no arguments are passed, the usage information is displayed. If no source files are passed (only one or more command switches) the program switches to console mode, and source files can then be typed in separately. Pressing ENTER adds a file to the list, pressing CTRL+Z and then ENTER sends that list to the program, which are then processed.

Source filenames without an extension are assumed to have the '.cg' extension. If specified, any extension will work just fine. You can also pass a list of files by preceding the file that has the list with the '@' symbol; i.e. cg @roomlist.txt will open the file roomlist.txt and read each line of the file as an input source file name. You can also specify DOS environment variables 'HEAD' and 'TAIL', which are added to the beginning and end of the file list passed on the command line.

Each source file's name (not extension) must include a number, or CG.EXE will throw an error. The output file for a source is always the sourcefile truncated at the first number, with the number as the extension. For example, 'logic1.cg' output is 'logic.1', 'log2a3.txt' becomes 'log.2', etc.

The maximum number of source files that can be compiled at one time (including those specified by name, by wildcard and in file lists) is 200, which seems odd since AGI allows up to 256 logics in a game.

CG.EXE processes each source file separately. The sequence of actions is:

    • parse input file name, build output file name
    • reset compiler parameters
    • open source file and assign to buffer space
    • create output file (overwriting any existing file) and assign to buffer space
    • initialize the hash table, adding predefined symbols
    • allocate space for output, and for AGI messages
    • compile the source (errors are displayed as they are encountered)
    • close the output file
    • display results, including total number of errors encountered
    • repeat for next source file

Feature Summary

Error detection in CG.EXE is limited to basic syntax checks. If it can create output without a major error (mismatched brackets, misspellings, etc.) it does quietly and continues. Even if an error is encountered, CG.EXE skips over the erroneous source an continues. It will then display a list of all found errors.

There are no error checks on the output. No errors or warnings are given if the compiled code would cause runtime errors. Programmers were expected to avoid errors by self-checking their code.

Sierra's compiler is intentionally designed to allow for changes/additions to the action and test commands without needing to rebuild CG.EXE itself. CG.EXE only manages the syntax used to access the commands; the commands themselves all need to be declared for every file that is compiled.

CG.EXE converts all unknown tokens (those that are not numbers or symbols) it encounters into a hash value (by summing the ascii values of all characters in the token, and then returning that value MOD 203). This value is then compared against entries in CG.EXE's token hash table to determine what it represents. If more than one token has the same hash value, CG.EXE creates linked lists for each hash value to avoid conflicts. If the number of tokens is too large such that all memory is used up to hold them, CG.EXE will throw an error and quit. Tokens are case sensitive.

On startup, the following tokens are added to the hash table: %include, %tokens, %test, %action, %flag, %var, %object, %define, %message, %view, #include, #tokens, #test, #action, #flag, #var, #object, #define, #message, #view, goto, if, else, FLAG, OBJECT, MSG, WORD, NUM, MSGNUM, VIEW, VAR, ANY, WORDLIST. These can be broken down into three groups:

    • Preprocessor symbols: The symbols starting with '%' or '#' are compiler commands that add symbols to the hash table which tell CG.EXE how to handle other symbols it will encounter later. Note that there are two versions of each, with either the '#' or '%' character to start. There is no difference between them; CG.EXE just offers the flexibility to use either format. The purpose and use of these tokens is explained in detail in the Sierra Syntax topic.
    • Keyword tokens: 'if', 'else' and 'goto' are used to control program flow.
    • Argument type tokens: 'NUM', 'FLAG', 'VAR', 'OBJECT', 'VIEW', 'MSGNUM' and 'WORDLIST' tokens are used to assign argument types to commands.

      • 'MSG' is legacy type, and should never be used, as no token will ever match it. It is likely left over from an earlier version of AGI.

      'WORD' is legacy type as well. It matches a single word from WORDS.TOK. In very early versions of AGI, there were separate versions of the said command that accepted only one or two words:

      %test said1(WORD) .....
%test said2(WORD, WORD) .....

      When the current said command was added to AGI, with the ability to have a variable number of words, the 'WORD' type became obselete so CG.EXE was revised to use the 'WORDLIST' type instead. But the 'WORD' type was not removed.

      The 'ANY' type matches any of the other types except WORD, WORDLIST. You can actually use it in a command declaration, and when compiling any type of argument will work:

      %action Increment(ANY) 1
%var v1 1
%flag f1 1
%object o1 1
%view vw1 1
Increment(1); [ OK
Increment(v1); [ OK; same as Increment(1)
Increment(f1); [ OK; same as Increment(1)
Increment(o1); [ OK; same as Increment(1)
Increment(vw1); [ OK; same as Increment(1)

      Its exact purpose is unknown. It doesn't show up in any source files for original Sierra games, and there is no documentation explaining what it is for. Although it can be used in source code without error, it should be avoided, as it is better to use strongly typed arguments.

In most cases, if an error is detected in the source code, CG.EXE will skip to the the end of the line and continue. This lets the compiler find multiple errors in a source file in one pass.

Bugs and Anomalies

There are a number of bugs in CG.EXE, as well as some unusual coding practices that can lead to unexpected results if specialized source code fragments are used ('hacks' or anomalies if you will). Some of the more interesting and potentially impactful ones are presented here, in no particular order.

    • Character '\x1a' is unusual in how CG.EXE handles it when it is encountered in a source file. When looking for a token, '\x1a' acts as end-of-file marker, meaning it stops reading data (as if it had reached the end of the file). Any additional text after '\x1a' is ignored.

      • However, there is a bug in how CG.EXE handles this character when building a literal string token (i.e. when a double quote mark is found). The internal read function in CG.EXE normally returns the ascii value of the next character it finds. But when '\x1a' is found, it returns '\xffff', which is supposed to trigger the end of input. When building the string, the return value is converted to a byte value ('\xff') and added to the string. This also means that if the last token in a file is a literal string without an ending double-quote, CG.EXE will read data beyond the end of the file looking for a closing double-quote, which almost always causes a 'message too long' error.

    • When goto commands use a destination label that has not yet been found in source code, CG.EXE builds a temporary array to track the proper location so once the label is found, the jump values can be inserted in the correct locations. CG.EXE has a limit of 19 jumps that can be unresolved. If more than that are in the source code, an error will be raised. Once the label is found, there is no limit on the number of goto commands that can use any given label.
    • The %define preprocessor allows any token to be re-defined. This means it is possible to re-define symbols ("==", "{", ";" etc.) and predefined keywords, preprocessors and argument types ("if", "#action", "FLAG", etc.) There is no practical use for this, and was probably never intended.
    • There is a major bug in how the '&' and '|' characters are handled. If CG.EXE's parser has already determined a token type and is checking additional characters, if the next character is '&' or '|', it gets put back in the queue, resulting in an infinite loop that freezes the program.
      • a || b && c [ does not crash CG.EXE
abcd||  [ no space between 'd' and '|' will crash CG.EXE
= & [ space good- will not crash
=&  [ no space bad- will crash CG.EXE
|&   &| [ both of these combinations will crash CG.EXE

[ a practical example - this looks like it should work:
if (v99 > 1&& v99 < 10) { [ NOT OK
[ but without a space between '1' and '&' it will crash CG.EXE
[ this line will compile:
if (v99 > 1 && v99 < 10) { [ OK

      The effect of this is that using '&' or '|' inside any token other than the "&&" and "||" operators will put CG.EXE in an infinite loop and freeze.

      Tokens can start with the '&' or '|' character, but the resulting token is considered a symbol, so it can't be used as a define name or any other purpose.

    • Unlike all other cases involving number tokens, the message number parameter of the %message preprocessor is NOT converted to a byte value. Instead, the 16 bit value is used. If it is greater than 255 or negative an error is raised. But if it is zero, (which should not be allowed) CG.EXE will use an invalid memory location when creating the message section of the logic which corrupts the output. If the value is negative, CG.EXE will raise an error, but no error is raised if the value is zero so the corrupted output is mistakenly treated by CG.EXE as a successful compilation.
    • CG.EXE will raise an error if the length of a message is greater than 999 characters. It is extremely unlikely that a programmer would intentionally use a message that's even a fraction of that. (The entire screen in AGI could hold 1000 characters- 25 rows of 40 columns.) This check was probably added to catch situations when an ending quote mark is missing.
    • If the last token of a logic is a literal string that is missing its end quote mark, CG.EXE will overflow the input buffer and read invalid data until an end quote mark is found or the 999 character limit is reached. If there happens to be a end quote mark in the invalid data, the message will include the extra characters from the invalid data.
    • CG.EXE allows multiple consecutive NOT symbols (!) in if blocks. This bug can lead to hard to find errors, since at runtime AGI toggles the 'NOT' status with each NOT token:
      • %flag f99 99
reset(f99);
if (!!isset(f99)) { [ extra '!'; same as 'if(isset(f99))'
  %message 1 "Flag 99 is reset."
  print( 1);
}

      In the example above, the message will never be displayed.

    • The 'not equal operator' "!=" is the only one that CG.EXE treats as two separate tokens, which means you can include white space between the characters. ('! =' is the same as '!='.) This was probably an oversight when CG.EXE was written.
    • Although '!=' is allowed as the 'NOT' version of '==', CG.EXE does not include 'NOT' versions of '<' and '>', i.e. there are no ">=" or "<=" operators ('!>' and '!<' don't work either).
    • There is a bug in the handling of the '@' character. CG.EXE treats it as a symbol character (so "@=" is correctly treated as a symbol). But if the character following the '@' is not '=' or a separator, CG.EXE will build a token that looks like an identifier, but is treated as a symbol:
      • %define @sometoken 99 [ NOT OK; name can't be a symbol
%define sometoken @99 [ OK; compiles but the token can't be used for anything

      This bug also applies to tokens beginning with '&' and '|'.

    • Because of a bug in number parsing, CG.EXE will skip any non-numeric characters that immediately follow the number without white space. The number value will use the number digits up until the first non-numeric character is found.:
      • %define anum 99abc [ same as '%define anum 99'
    • In alternate test syntax tokens declared as %object can be used where the argument type is a number. This is useful when testing reserved variable v5 (Object Edge Code). However, it does not work when using normal test command syntax.
      • %object ego 0
if (v5 == ego) [ OK; same as 'if (v5 == 0)'
if (equaln(v5, ego)) [ NOT OK; wrong argument type
    • In what appears to be a bug, the symbol tokens '(', ')' and '{' are ignored when CG.EXE is expecting a command or keyword. These tokens should not be allowed in this situation.
    • In CG.EXE's handling of the ':' character to locate labels, when a ':' character is found CG.EXE sets the token type to a label, then gets the next full numeric or identifier token, as long as it is on the same line. If nothing else is on the line, a null value is assigned to the label, but no error is raised. If the token is numeric, it will be added as a label name. But since null and numeric tokens cannot be used in the goto statement, null and numeric labels can never be used.
    • Do not try to %define a colon, (%define colon :). The define will work, but if you try to use it CG.EXE will crash.
    • If the parameter of a goto statement is a defined name, CG.EXE confuses the define data as label data, which leads to invalid jump values. This bug will lead to corrupt logic code, so programmers should be careful to avoid using a define name in a goto statement.
    • If the source file ends with an if or else block that is missing its closing curly brace '}', CG.EXE provides a warning, but doesn't consider it as an error. If the missing brace is part of an if block, the jump value assigned to the if block will not be updated with the correct value.
    • If an else statement is found that doesn't immediately follow the closing curly brace of an if block, CG.EXE does not raise an error. Instead, it treats the else as a goto, where the jump-to location is set to the closing curly brace of the else block. This appears to be a bug.
    • If CG.EXE attempts to process a said command before WORDS.TOK is loaded with the %tokens preprocessor, instead of raising an error, it will enter an infinite loop and lock up.
    • When processing alternate action syntax and alternate test syntax statements, CG.EXE actually inserts the matching command token into the data stream, as if it had been typed in the source code and then compiles that symbol. This means that commands with alternate syntax must be declared with the exact spelling that CG.EXE uses when it inserts them.
      • v99 = 1; [ CG.EXE changes this to 'assignn(v99, 1)'

      If the declared command is not spelled exactly the same, a 'command not found' error is raised.
