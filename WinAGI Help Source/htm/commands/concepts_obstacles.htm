Managing Obstacles

There are six different types of obstacles that can restrict a screen object’s movement:

    • Control lines on the priority screen
    • Blocks set by the block command
    • Land or water, if the object is restricted to either
    • The horizon
    • The screen edges
    • Other objects

During each interpreter cycle, as well as when positioning or repositioning objects, AGI checks to see if each screen object has encountered an obstacle. When checking for obstacles, none of pixels on the object's baseline (bottom row of pixels) are allowed to come in contact with an obstacle that is being checked. The height of an object has no effect on obstacle detection.

When an object collision with an obstacle is detected, AGI will prevent the object from moving, but it does NOT set the object's direction to 'stopped'. The object will still try to move each interpreter cycle. Also, AGI does not automatically stop an object from cycling when it fails to move due to encountering an obstacle. The programmer must write code to manually stop cycling in this case.

 

Control lines on the priority screen

On the priority screen, the values 0 and 1 represent unconditional and conditional control lines respectively. No object is ever allowed to be positioned with its baseline on an unconditional control line. Objects that are observing blocks are not allowed to be on parts of the screen with a priority of 1 (conditional control line). The ignore.blocks command (or setting priority to 15 with the set.priority command) can be used to allow objects to pass over conditional control line pixels. The observe.blocks command restores normal behavior.

Using the set.priority command with a value of 15 will also cause an object to ignore conditional control lines, overriding any use of the observe.block command.

Note that the condition that AGI tests for is if the object's new position is on top of a control line pixel; if an object's step size is large enough, it is possible for the object to 'jump' over control lines. When drawing priority screens, you may need to widen your control lines, or add extra lines to deal with this possibility.

When object positions are updated for movement during the interpreter cycle, if the new location of an object would put it on a control line, AGI will return it to its previous position. This does not change the object's direction to 0, nor does it cause cycling to stop.

Blocks

Using the block command, you can add an invisible rectangle to the screen that acts similarly to a conditional control line. The bounding rectangle is not actually drawn on the priority screen, but it behaves as a single pixel wide set of lines forming the rectangle defined by the coordinates passed to the block command. AGI will not allow screen objects to cross in to or out of the bounding rectangle, unless they have been issued the ignore.blocks command. Use the observe.blocks command on objects that are ignoring blocks to begin observing the block again. Use the unblock command to remove the block.

Using the set.priority command with a value of 15 will also cause an object to ignore blocks, overriding any use of the observe.block command.

Unlike control lines, ONLY the left bottom pixel of the object is checked, not the entire width of the object's baseline.

The block feature is basically a 'fence' for screen objects, to either restrict their movement to a rectangular area, or to prevent them from entering a rectangular area. If an object is outside the bounding rectangle, AGI will allow it to move onto the bounding rectangle line, but not inside the bounding rectangle. Objects inside the bounding rectangle cannot move onto the bounding rectangle line. This means that if the bounding rectangle's height or width is only one or two pixels, the block won't actually work; in this case the object would always appear 'outside' or 'on' the block's bounding rectangle, and would never be able to be 'in' the rectangle.

Also, if the object's step size is big enough, it can 'jump over' the block; as long as the new position is also outside the bounding rectangle. Keep in mind that the object can never cross from inside to outside (or outside to inside) the bounding rectangle no matter what the step size is.

In version 2.089 and version 2.272, when a block is encountered while an object is in normal motion, the object will not move, but the object's direction remains unchanged (similar to the behavior for encountering control lines). If the block is removed, the object will resume moving. In all other later versions, when a block is encountered, the object's direction is set to zero, and the object will not resume moving after the block is lifted unless the object's direction is reset. This is different than the behavior for encountering control lines.

Land and Water

Pixels on the priority screen with a value of 3 are considered "water". Other areas are considered "land". The default behavior for objects is no restrictions; the object can be on land or water. The terms water and land are for labeling purposes only; priority 3 areas could be used in a game to represent any area where the programmer wants to restrict or prohibit movement, such as ice, lava, sand, etc.

After calling the object.on.water command, an object is restricted to pixels with priority of 3. The object will not be allowed to move if any part of its baseline would be on non-water pixels, in other words, the entire baseline must be on water. If the object is not on water when this command is called, AGI will use the shuffling function to move it to water on next interpreter cycle update. If there is no space on water large enough for the object, AGI will freeze, so make sure you don't call this command when there are no water areas present.

The object.on.land command works slightly differently than object.on.water. Instead of restricting the object to all land pixels, as long as at least one pixel of the object's baseline is on land, the position is allowed.

Do NOT use both commands on the same object at the same time; this will cause AGI to freeze, as it will be unable to meet both conditions simultaneously. To remove all restrictions, use the object.on.anything command.

The ego object cannot be restricted by these commands; AGI resets its permissions in every interpreter cycle, so trying to use object.on.water and object.on.land on ego has no affect. Although it can't be restricted, you can track ego's presence on water; whenever ego’s baseline is completely on water, reserved flag f0 (ego on water) is set (it is cleared as soon as ego moves out of the water).

Screen Edges

Objects are not allowed to go past any screen edge. If the ego reaches a screen edge, reserved variable v2 is assigned a code to indicate which edge. Note that this check accounts for the width and the height of the object, not just the x and y position (which the location of the lower left pixel of the object).

If an object other than ego reaches an edge, its index number is stored in reserved variable v4 (object touching edge) and reserved variable v5 (object edge code) is assigned the edge code. If more than one object reaches an edge at the same time, the object with the highest number will be stored in these variables.

The Horizon

The horizon is an invisible horizontal line, usually near the top of the screen, which works similarly to the right/left/top screen edges. The horizon represents the furthest away (up the screen) that object can move before being considered 'off screen'. When an object touches the horizon, reserved variables are assigned an edge code as described above. The edge code is the same when an object hits the horizon or top edge; AGI treats both cases the same.

Normally, objects can not go above the horizon. This can be toggled with the ignore.horizon and observe.horizon commands.

Objects cannot 'jump' over the horizon, no matter how big the object's step size.

Other Objects

By default, object collisions are normally not allowed- that is, the baselines of objects are not allowed to touch each other. This can be changed, however, with the ignore.objs command. Default behavior can be restored with the observe.objs command.

The actual function that checks for object collision first compares the X components of the objects' baselines. If the baselines don't overlap (i.e. one object's baseline has X values that are less than all X values of the other object), then there is no collision.

If the objects X coordinates do overlap, AGI then checks if the Y values are identical. if they are, then AGI treats it as a collision. If not, a final check is made comparing the two objects' PREVIOUS Y values. If the Y values have swapped (meaning they probably 'jumped' over each other) then AGI treats it as a collision.

There is a bug in all AGI versions in how the baselines are measured; the rightmost pixel of an object is equal to the object's x value, plus the object's width, minus 1. The function does not do include the minus 1 calculation though. This means AGI thinks objects overlap if one object is exactly adjacent to another (i.e. no gap between them). So objects can't move right next to each other (they will always have a gap of one pixel between them) and if repositioning, attempting to place one object adjacent to another will force it to 'shuffle'.

When a collision is detected during movement, the object being tested is moved back to its previous position. When a collision is detected when an object is first drawn or is repositioned, the object gets 'shuffled'.

'Jump' Check

In addition to preventing baselines of two objects from touching, AGI also checks to see if objects have 'jumped' over each other when updating positions in each interpreter cycle or when doing a reposition.

The jump check only checks in the vertical direction, that is, if both objects have the same Y value, as long as the new positions don't overlap, AGI will allow one object to jump over the other in a horizontal direction. Also keep in mind that for the jump check, ignoring objects applies independently; in other words if object A is ignoring other objects, it can 'jump' over object B, but object B cannot jump over object A.
