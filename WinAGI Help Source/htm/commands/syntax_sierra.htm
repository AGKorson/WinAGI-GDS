AGI Logic SIERRA Syntax

The details of Sierra's original syntax for AGI were not determined until after the fan syntax was established. It wasn't until Sierra's logic compiler CG.EXE was fully disassembled that original Sierra syntax was fully understood.

Syntax Overview

Sierra syntax is also based on the C family of languages. Unlike Fan syntax, the command names are not predefined; each command must be declared before it is used in the source code.

There are fewer argument types in Sierra syntax, but the the option to define substitutes for arguments is the same. As with Fan syntax, all commands and tokens are case sensitive.

Sierra's logic compiler CG.EXE has several bugs that allow for some unusual/unexpected results in certain situations. To maintain full compatibility with Sierra's compiler, WinAGI mimics these results, but provides warnings an/or error messages as appropriate.

Tokens

A token is defined as a group of one or more characters separated by a separator character or another symbol token. Characters are limited to byte values; the source code text is converted to original MSDOS (code page 437) encoding prior to compiling. Tokens are case sensitive.

The compiler runs a single pass through the source code, processing tokens to determine how to convert source code text to byte code output.

The compiler recognizes five token types: separators, symbols, numbers literal strings and identifiers. The first character of a token determines its type.

Separator Tokens

The space ' ', tab '\x09', carriage return ('\r' or '\x0d') and new line ('\n' or '\x0a') are the primary separator characters. When looking for the start of a new token these characters are skipped. They also mark the end of a token.

The '[' character marks the start of a comment. When looking for the start of a new token this token all characters following a '[' are skipped until a new line '\x0a' character is found.

To maintain compatibility with Sierra's logic compiler CG.EXE, non-printing character '\x1a' is treated as an 'end-of-file' character. If encountered when looking for the start of a new token, compiling ends immediately and the rest of the source code is ignored.

Symbol Tokens

If the first character of a token is one of these characters:

!&()+,-:;<=>@{|}

the token is a symbol. Most symbol tokens are just a single character. However, the compiler will check the next character and return the following as two-character tokens:

++, --, +=, -=, ==, @=, =@, &&, ||

In most cases, the compiler does not evaluate the character after identifying a one or two character symbol token. This means symbol tokens also act as separators. The '@', '&' and '|' characters are exceptions. If the start character is '@', '&' or '|' and is not the "@=", "&&" or "||" token, then compiler will add number and identifier characters to the token until a separator or symbol character is encountered. The resulting token is still treated as a symbol. This means it is not possible to have identifier tokens start with the '@', '&' or '|' characters.

Another exception is the '-' character. If the next character is a digit, the token type is changed to a number and processed as described below, returning a negative number.

There is a bug associated with the '&' and '|' characters; if not part of a "&&" or "||" token, adding a '&' or '|' character to any other token (unless inside a literal string or comment) will cause the compiler to enter an infinite loop and freeze. To maintain compatibility with Sierra's logic compiler CG.EXE, the characters '&' and '|' are not allowed in any token except "&&" and "||" and literal strings. Any other use will raise an error.

Number Tokens

If the first character of a token is a digit (0-9), or if it is a '-' and the next character is a digit, the token is a number. The compiler will add characters to the token until a separator or symbol character is found. The token is converted into a value by reading digits off the token until a non-number character is reached. The value is converted to a two byte signed integer, then only the lower byte is kept.

123         [ number token with value of 123
234abc567   [ number token with value of 234
1.001       [ number token with value of 1
256         [ number token with value of 0
-1          [ number token with value of 255

The message number parameter in the %message preprocessor statement is the one exception- the message number is not converted to a byte value before it is validated.

Literal String Tokens

If the first character is a quote mark '"' the token is a literal string. Literal strings are only used as the message text parameter of the %message preprocessor. The token will include all characters up to the next non-escaped '"' character.

Line breaks inside the literal string are ignored, which make it easy to split the message text over multiple lines for readability:

%message 4 "You
 can't
 do
 that
 now."
[ same as '%message 4 "You can't do that now."'

To maintain compatibility with Sierra's logic compiler CG.EXE, '\x1a' characters inside a literal string are converted to character '\xff'.

Identifier Tokens

A token starting with any other character (includes letters A-Z and a-z, symbol characters #$%'*./?\]^_`~ , extended characters (>= '\x7f') and non-printable characters <= '\x1f' (except '\x0a', '\x0d' and '\x1a')). The token will include all characters up to the next separator or symbol token. Character '\x1a' acts as a separator character.

In most cases, the identifier token is then checked against the defines list and replaced with type and value of the matching define. However the following situations will NOT convert an identifier token:

    • %include filename parameter
    • %tokens filename parameter
    • goto destination
    • %action or %test name parameter
    • %define, %object, %var, %flag, %view name parameter
    • said word arguments

Keyword Tokens

There are only three built-in keywords: if, else, and goto.

    • if
    • else
    • goto

Conditional Statements (if and if ... else blocks)

An if block will execute one or more commands only if the condition of the if statement is TRUE. The test commands must be enclosed in parentheses '()', and the action commands to be executed must be enclosed in curly brackets '{}':

if (<test commands>)
  {
  <action commands>
  }

Multiple tests within the same if statement must be separated by && (representing 'AND') or || (representing 'OR'). When test commands are to be ORred together, they must be placed in parentheses. If test commands are to be ANDed together, they must not be placed in parentheses:

if((<test1> || <test2>)) [  ORed tests in parentheses
  ......

if(<test1> && <test2>)  [  ANDed tests not in parentheses
  ......

if(<test1> && (<test2> || <test3>)) [ combination of ANDed and ORed tests 
  ......

if((<test1> && <test2>) || <test3>)   [ NOT OK; ORed tests are not in parentheses

An if ... else block is similar to the if block, but will execute one block of action commands if the if statement is TRUE, and a different block of action commands if the if statement is FALSE:

if (<test commands>)
  {
  <action commands if test is TRUE>
  }
else
  {
  <action commands if test is FALSE>
  }

Normally, the else keyword is used as described above, in conjunction with the if keyword. However, Sierra syntax allows the else keyword to be used as a substitute for the goto keyword, if the else keyword does not immediately follow the closing curly brace of an if block:

<action commands>
else  [ ALLOWED - else command that does not follow an if block
  {
    <skipped action commands>
  }

[ equivalent code:
<action commands>
goto label1
    <skipped action commands>
:label1

Although this use of else is allowed, it should be avoided as it is bad coding practice.

goto Statements

The goto keyword is used to transfer the processing of commands to a new location in the logic script. It takes a single parameter, the name of a label. The label must NOT be enclosed in parentheses:

goto Label1   [ OK

goto(Label2)  [ NOT OK

Labels are 'markers' that are used by the goto command to control program flow. The label must begin with a single colon character. Unlike Fan syntax labels cannot end with a colon. White space is allowed between the colon and the label name, but they must both be on the same line, or a null label will be created.

:Label1 [ OK
Label2: [ NOT OK
: Label3 [ OK
:
Label4 [ NOT OK

There is a limit of 50 labels that can be added to a single logic. More than that will cause an error.

Preprocessor Tokens

In addition to keywords, there are also ten 'pre-processor' tokens. These tokens start with '%' or '#' and are compiler commands which tell the compiler how to handle other tokens it will encounter later. Note that there are two versions of each, with either the '#' or '%' character to start. There is no difference between them; the compiler just offers the flexibility to use either format.

    • %include or #include
    • %tokens or #tokens
    • %action or #action
    • %test or #test
    • %flag or #flag
    • %var or #var
    • %object or #object
    • %view or #view
    • %message or #message
    • %define or #define

Including files

You can include (embed) another file in your logic source code by using the #include or %include token. The filename of the file to be included follows the command, and must be enclosed in quotation marks.The filename can include path information, including relative path notation. Wildcard characters are not supported. The compiler requires the backslash character to be 'escaped', i.e. use "\\" to represent a single '\'. If the file does not include path information, it is assumed to be in the same directory as the source code file:

%include "file1.txt"
%include "c:\\documents\\file2.txt"
%include "..\\file3.txt"
%include "c:\file4.txt"  [ NOT OK
%include "*.txt"  [ NOT OK

Included files can be nested (for example "file1.txt" can include "file2.txt" and "file2.txt" can include "file3.txt" etc.) but there is a limit of 5 layers of nesting. More than that will cause an error.

Referencing WORDS.TOK file

The compiler requires a reference to the game's WORDS.TOK file to be able to process said commands. The #tokens or %tokens preprocessor token takes a path to the file as the only argument:

%tokens "..\\WORDS.TOK"

The filename doesn't have to be WORDS.TOK, but it must be a valid AGI word file. It must be enclosed in double-quotes. It can include path information, but not wildcards. The compiler requires the backslash character to be 'escaped', i.e. use "\\" to represent a single '\'. If the file does not include path information, it is assumed to be in the same directory as the source code file.

If you do not assign a WORDS.TOK file with the %tokens preprocessor token, the compiler will raise an error when it tries process a said command.

Action Command Tokens

Action commands represent the user program functions that are available in AGI. Each action command is converted to the appropriate byte value that gets stored in the logic's compiled pseudo-code. Action commands are specified by the command name followed by parentheses which contain the arguments, separated by commas. The parentheses are required even if there are no arguments. Action commands have no return value.

Unlike Fan syntax, action command names are not predefined; each command must be defined using the #action or %action token:

%action addn(VAR, NUM) 5
%action cancel.line() 138

The first parameter is the desired name of the action command, followed by argument types in parentheses separated by commas. The last parameter is the bytecode value of the command to be assigned when the action command is used. If there are no arguments to the command the parentheses are still required.

The number of arguments in the command declaration must match what the interpreter uses:

%action addn() 5  [ NOT OK; addn command needs two arguments

The argument types used should match the actual argument types that the command uses at runtime. For example, command 5, the 'addn' command, should use (VAR, NUM) as argument types, since the command adds a numeric value to a variable.

As long as the number of commands match, though, you can technically use any argument type (except WORDLIST) for any of the arguments in the command:

%action addn(VIEW, OBJECT) 5  [ OK, but crazy!
%view   a_view     5
%object an_object 10
addn(a_view, an_object); [ OK; compiles without error

Although the example above will compile, and when run in AGI it will actually add 10 to variable 5, there is little practical value in using constructs such as this.

Alternate Action Syntax

Substitutions for the following action commands are available:

increment(v30);       ++v30;
decrement(v30);       --v30;  
assignn(v30,4);       v30 = 4;
assignv(v30,v32);     v30 = v32;
addn(v30,4);          v30 += 4;
addv(v30,v32);        v30 += v32;
subn(v30,4);          v30 -= 4;
subv(v30,v32);        v30 -= v32;
lindirectn(v30,4);    v30 @= 4;
lindirectv(v30,v32);  v30 @= v32;
rindirect(v30,v32);   v30 =@ v32;

Note that there are significant differences between Fan syntax and Sierra syntax regarding alternate syntax options:

    • increment/decrement have only one allowed form
    • addition and subtraction don't allow the long form
    • no substitute for multiplication/division commands
    • indirection uses a different assignment operator

Importantly, when using alternate action syntax, the substituted commands must be declared with exactly the same name as the Fan syntax version:

%action AddN(VAR, NUM) 5  [ not the same as 'addn'
v30 += 4; [ NOT OK; command 5 must be declared as 'addn'

Test Command Tokens

Test commands return a value of true or false are only used with the if keyword. As with action commands, arguments for the command must be enclosed in parentheses, separated by a comma, and the parentheses are required even if there are no arguments for the command.

Unlike Fan syntax, test command names are not predefined; each command must be defined using the #test or %test token:

%test equaln(VAR, NUM) 1
%test isset(FLAG) 7
%test have.key() 13

The first parameter is the desired name of the test command, followed by argument types in parentheses separated by commas. The last parameter is the bytecode value of the command to be assigned when the test command is used. If there are no arguments to the command the parentheses are still required.

The number of arguments in the command declaration must match what the interpreter uses:

%test equaln() 1  [ NOT OK; equaln command needs two arguments

An exclamation point (!) placed in front of a command signifies a NOT, which toggles the result of the test command:

if (!isset(f7)) ......
if (!have.key()) ......

Alternate Test Syntax

Substitutions for the following test commands are available:

equaln(v30, 4)        [ v30 == 4
equalv(v30, v32)      [ v30 == v32
greatern(v30, 4)      [ v30 > 4
greaterv(v30, v32)    [ v30 > v32
lessn(v30, 4)         [ v30 < 4
lessv(v30, v32)       [ v30 < v32
!equaln(v30, 4)       [ v30 != 4
!equalv(v30, v32)     [ v30 != v32

Flags can be tested by just using the name of the flag, as an alternative to the isset command:

if (f6) .....  [  same as if (isset(f6))

Variables can be tested as either TRUE (> 0) or FALSE(== 0) by just using the name of the variable:

if (v0) .....  [  same as 'if (greatern(v0, 0))' or 'if (v0 > 0)'

Note that there are several differences between Fan syntax and Sierra syntax regarding alternate syntax options. First of all, in Sierra syntax the NOT operator is allowed before an alternate syntax test:

if (!v30 == 4) ..... [ same as 'if (!equaln(v30, 4))' or 'if (v30 != 4))'
if (!f6) .....  [  same as if (!isset(f6))
if (!v30 > 4) .....  [  same as if (!greatern(v30, 4))
if (!v0) .....  [  same as 'if (!greatern(v0, 0))'

Unlike Fan syntax, there is no alternate form for negated versions of greatern, greaterv, lessn and lessv:

if (v30 <= 4) ..... [ NOT OK
if (!v30 > 4) .....  [  OK

In alternate test syntax only, tokens declared as %object can be used in test commands that use the NUM type argument. This works in alternate syntax only, not in the regular command syntax:

%object ego 0
if (v30 == ego) .....  [  OK; same as 'if (v30 == 0)'
if (equaln(v30, ego) .....  [ NOT OK

When negating alternate forms of equaln and equalv, white space is allowed between the NOT operator and the '=' symbol:

if (v30 !     = 0) .....  [ OK in Sierra syntax

The said Command

The said test command is the only AGI command that uses the WORDLIST argument type. The number of arguments in a said command can vary (but must be at least 1). The arguments are words from the WORDS.TOK file, without quotes:

if (said(look)) .....
if (said(open, door)) .....

If a word includes a space character, use the '$' character in place of the space:

if (said(look$at)) .....

Unlike Fan syntax, word group numbers are not allowed as arguments in said command:

if (said(4, 80)) .....   [ NOT OK

Flag Tokens

FLAG arguments require a token that is declared with the %flag or #flag preprocessor token. The first parameter is the desired flag name, and the second parameter is the byte value of the flag assigned to the token.

%action set(FLAG) 12
%flag flag99   99
set(flag99);

%var var99   99
set(var99); [ NOT OK; wrong argument type
set(99); [ NOT OK; also wrong argument type

Variable Tokens

VAR arguments require a token that is declared with the %var or #var preprocessor token. The first parameter is the desired variable name, and the second parameter is the byte value of the variable assigned to the token.

%action assignn(VAR, NUM) 3
%var var99   99
assignn(var99, 4);

%flag flag99   99
assignn(flag99, 4); [ NOT OK; wrong argument type
assignn(99, 4); [ NOT OK; also wrong argument type

Object Tokens

OBJECT arguments require a token that is declared with the %object or #object preprocessor token. The first parameter is the desired variable name, and the second parameter is the byte value of the object assigned to the token.

%action animate.obj(OBJECT) 33
%object ego 0
animate.obj(ego);

%flag flag99   99
animate.obj(flag99); [ NOT OK; wrong argument type
animate.obj(0); [ NOT OK; also wrong argument type

Original Sierra syntax used the term 'object' to refer to both screen objects and inventory items. When programming in Sierra syntax it is up to the programmer to recognize the difference and make sure the correct arguments are used in commands.

%action animate.obj(OBJECT) 33
%action get(OBJECT) 92
%object pirate 1
%object dagger 2
animate.obj(pirate);
get(dagger);

[ this will work but probably is not what was intended
animate.obj(dagger);
get(pirate);

Unlike Fan syntax, inventory items cannot be referenced by the item name in quotes:

%action get(OBJECT) 92
get("dagger"); [ NOT OK

View Tokens

%view or #view:

VIEW arguments require a token that is declared with the %view or #view preprocessor token. The first parameter is the desired view name, and the second parameter is the byte value of the view assigned to the token.

%action load.view(VIEW) 30
%view deadEgo   23
load.view(deadEgo);

%flag flag99   99
load.view(flag99); [ NOT OK; wrong argument type
load.view(23); [ NOT OK; also wrong argument type

Message Text Declarations

MSGNUM arguments require a token that is declared with the %message or #message preprocessor token. Unlike other preprocessor tokens, there is no token name parameter. The first parameter is the byte value of the message and the second parameter is the literal text string to assign to the message number. To access the token in a command, use the number assigned. You cannot use the literal string as an argument.

%message 4 "You can't do that now."
%action print(MSGNUM) 101

print( 4);  [ OK
print("You can't do that now.");  [ NOT OK
print( 5);   [ NOT OK; message 5 not declared

To maintain compatibility with Sierra's logic compiler CG.EXE, the value of the message number parameter is NOT converted to a byte value. The actual number value must be >= 1 and <= 255.

Also, note that there is no message 0; message numbers start at 1. Attempting to declare a number zero message will cause an error. When assigning numbers, you are not required to assign them sequentially. The compiler will automatically assign a null pointer for any messages that skipped over and not explicitly assigned.

Messages are also limited to 999 characters. If a message exceeds this, an error is raised. This condition most likely means there is a missing ending quote mark for the message.

Compiler Escape Character and Special Characters

To include certain special characters, the compiler uses the backslash '\' as an 'escape' character. Valid compiler escape codes include:

    • \n - Inserts a 'new-line' character (ascii code '0x0A')
    • \" - Inserts a double quote character (ascii code '0x22'); needed because the compiler normally treats a double quote character as a string delimiter
    • \\ - Inserts a backslash character (ascii code '0x5C');

      If the character following the escape character '\' is not one of the above, the escape character '\' is ignored, but the following character is kept.

      Because AGI also treats the backslash as a runtime escape character, there are several situations that need special sequences to display certain characters:

        • \\\\ - Inserts a single backslash when printing or displaying text; when the string is compiled, the four backslashes become two backslashes in the actual message text used by AGI, and then when AGI draws the message text on the screen it interprets the first backslash as the runtime escape character, and then displays only the second backslash.
        • \\% - Inserts a percent sign (ascii code '\x25') when printing or displaying text; normally when AGI encounters a percent sign, it treats it as a special format code. To display a percent sign, it must be preceded by a runtime escape code (requiring "\\" in the source code)
        • "\\ " - Inserts a non-breaking space. When AGI formats a message to display when using print commands, it automatically inserts line breaks between words so text does not extend past the print area margins. It will not insert a line break between words separated with a non-breaking space.

      Within a message text argument, only standard ASCII characters (1-255) may be used. Characters above 127 and less than 32 are not fully supported by AGI, but can be used if the programmer understands the restrictions and limitations. Most AGI development tools do not provide a means to easily add these characters, although WinAGI does.

      Generic Defined Tokens

      To improve the readability of source code and to provide flexibility, any number, symbol or previously declared token can be assigned a name using the #define: or %define preprocessor token. The first parameter is the desired token name and the second parameter is the number, symbol or token that it will represent.

      %define ONE    1
%var v100   100
%define temp    v100

      You can also 're-define' a token:

      %var v200   200
%define lvar1  v200
%define counter  lvar1
addn(counter, 1); [ same as addn(lvar1, 1) or addn(v200, 1)

      Symbols can be nested in this manner as deep as you want. As long as each symbol is defined before it is referenced, the compiler will continue substituting define values until it reaches a predefined token type (NUM, FLAG, VAR, etc.)

      Define values can also be literal strings that you can assign to messages:

      %define m1 "a message"
%message 1 m1

      Define values can also be keywords, preprocessor tokens, argument type token or symbols. Although there is little practical value, it does allow constructs such as:

      %define #DEF #define
#DEF IF if
#DEF THEN {
#DEF ENDIF }
IF (isset(f1)) THEN
    ++v1;
ENDIF

      Argument Types

      When declaring action and test commands, the command's arguments must be designated as one of seven different argument types. When compiling the command, the compiler expects a token with a type that matches the designated argument type.

        • NUM: Use this argument type when you want a command to use a numeric argument.
        • FLAG: Use this argument type when you want a command to use a flag argument.
        • VAR: Use this argument type when you want a command to use a variable argument.
        • VIEW: Use this argument type when you want a command to use a view argument.
        • OBJECT: Use this argument type when you want a command to use an object argument. Note that he OBJECT argument type is used in original Sierra syntax for both screen objects and inventory items. Care must be taken by the programmer to keep them straight, because the compiler won't do it for you.
        • MSGNUM: Use this argument type when you want a command to use a message number as an argument value.
        • WORDLIST: This argument type is only used by the said command. It acts as a placeholder for one or more words from the WORDS.TOK file.

      There are three additional argument types that the compiler recognizes but you should NOT use them in command declarations: ANY, MSG and WORD. They are only included as defined types in WinAGI to maintain compatibility with Sierra's logic compiler CG.EXE.

      Command Terminators and Argument Separators

      Unlike Fan syntax, a terminating ';' character is not required after action commands. Using it is a good coding practice, but the compiler always ignores the ';' between action commands.

      equaln(v99, 1);
equaln(v99, 2) [ OK; ';' not required
; equaln(v99, 2); ; ; [ OK ';' characters are ignored

      In Sierra syntax, commas and semi-colons are treated exactly the same. For example

      addn(v1, 2);
addn(v1; 2),

      will both compile correctly.

      Comments

      In-line comments are set off by a single open square bracket character '['. All characters are ignored until a new line character or the end of file is reached.

      addn(v99, 1); [ Rest of line is ignored

      Sierra syntax does not support the double-slash comment marker (//) or block comments (/* ... */).
