Object Priority

To simulate a 3-D environment, AGI assigns all objects a priority. The background picture includes a separate priority screen that is used in conjunction with objects' priorities to create the illusion of 3-D.

By default, objects get assigned a priority automatically based on their vertical position. Objects with a higher priority are displayed on top of objects with a lower priority. Different parts of a picture can also be given different priorities. So something in the far distance would have a priority of 4 (everything else would go on top of this) and something right at the front of the screen would have a priority of 15 (nothing will go on top of this except objects with a priority of 15).

To set an objectĺs priority so it will remain constant wherever the objects is on screen, use the set.priority or set.priority.v commands. To change it back so the priority is dependent on the objectĺs position, use the release.priority command.

When two or more objects overlap, the one with the highest priority is drawn on top. If they have the same priority, the object with the higher y value (closest in 3-D) is drawn on top. If they have the same y value then the one with the higher index number is drawn on top.

Objects that have a fixed priority are assigned a virtual Y value so they can be drawn in the correct order with all other objects.

The set.pri.base command, added in version 2.936 included a bug in the code that determines the virtual Y value that would always return a value of 168, so the object would always draw on top of other objects with the same priority. The error only shows up when using a custom priority base with an object that has a fixed priority and another object without fixed priority is in the same priority band. The bug was fixed in version 3.002.098.

An objectĺs priority can be determined using the get.priority command.

Avoid Invalid Priority Values

If you use an invalid value in a set.priority or set.priority.v command, you will get unexpected behavior. The algorithm that AGI uses to draw an object uses only the lower nibble (i.e. lower four bits, i.e., PRIORITY(actual)=PRIORITY(value) & 0x0F), for setting the priority value for the object. But there is a bug in the function that AGI uses to determine the order in which to draw objects. Objects with a fixed priority are assumed by AGI to have an 'effective' Y value based on the priority value, but AGI's algorithm for determining the order based on Y values has a bug which uses the whole byte. The effect of this is that objects with priority values > 22 won't draw on screen at all.
