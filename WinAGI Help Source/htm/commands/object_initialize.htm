Initializing Objects

All screen objects (including ego) must be initialized before they can be manipulated and used in a game. There are four basic steps that must be followed:

1. Animate the object.

The animate.obj command initializes the object, setting up AGI to use it in subsequent commands. The number of objects is limited by the Maximum Screen Objects value taken from the game's OBJECT file. This value cannot be changed at runtime, but can be adjusted during game creation. More objects means more calculations during each interpreter cycle, which slows down AGI (this is not a significant concern on modern computer systems though).

2. Position the object on screen.

The default values for the object's coordinates are (0,0). Set the initial position of the object with the position or position.v commands.

3. Assign a view/loop/cel to the object.

The visual image used by the object to actually display it on screen comes from a view resource. A view resource is assigned to the object with the set.view or set.view.v command. You should assign the view before any other actions other than positioning are taken on the object. You must have the view loaded in memory (load.view or load.view.v) before you assign it.

By default, the loop and cel of a new object are zero. If you need to assign a specific loop and/or cel other than 0/0 to an object, you should do that AFTER positioning it. This is because when a loop or cel is assigned, AGI will validate that the dimensions of the newly assigned cel fit on screen. Since a newly intialized screen object has a position of (0, 0), AGI will adjust the y value so the object doesn't overlap the top edge and set the internal 'repositioned' flag and your object will not move on the next interpreter cycle. Positioning after this happens won't reset the internal 'repositioned' flag. While this is usually not noticeable, it can cause unexpected results if you are relying on precise location/movement of an object within your logic code. (Note that if the cel is exactly one pixel high, this discussion wouldn't apply.)

It is possible to assign the same view to more than one object. This can be used to create 'herds' of objects. The alligators in the King's Quest I moat are a good example.

4. Draw the object.

This is done using the draw command. The object will not be visible until it is drawn to the screen. If you later need to remove it from the screen, use the erase command. When drawing objects, the object's location (set by position or position.v) is validated using a method called 'shuffling' to ensure the object is not drawn outside the playable area, on a control line or on another object (if the object is not ignoring other objects).

Updating Property

Objects are automatically updated in each interpreter cycle while the graphics display is active. Updating means the objects are both moved and cycled according to their settings. The updating property is enabled by default when objects are initialized. The updating property of an object can be changed by using the stop.update and start.update commands.

When updating is disabled, the object will not move or cycle automatically, and all non-updating objects get drawn before all updating objects.

You can manually change its position or displayed cel by using the appropriate commands. However, it will not update on the screen unless its update property is set, or if a force.update or overlay.pic command is used.

Because of the way that AGI handles background bitmaps behind objects, changes made to a non-updating object will result in 'ghost' images if any command that affects heap memory is used..

When updating is enabled, cycling can be independently enabled or disabled by using the start.cycling and stop.cycling commands. See the Cycling Objects topic for additional information on cycling.

Keep in mind that the draw command always sets an object's update property to TRUE, so when you are initializing an object that you want updating disabled, you must call the stop.update command AFTER the draw command.
