Logic Compiler Warning 5050

Setting PRIORITY to < 4 or > 15 may cause unexpected results

AGI priority values should be in the range 4 to 15. Using a priority value outside this range may have unexpected results.

When priority is set to 0, 1 or 2 (values that correspond to control lines on the priority screen), AGI will not raise a runtime error, but the object will not be displayed on the visual screen. With the add.to.pic command, if you also specified a value for MARGIN (to draw a box around the base of the object), AGI will draw the box even though it won't draw the object.

Setting a priority value of 3 will only display pixels that are on water (i.e. the corresponding pixel on the priority screen also has value of 3).

There are two issues that arise from using values of priority greater than 15. First of all, the algorithm that AGI uses to draw an object uses only the lower nibble (i.e. lower four bits, i.e., PRIORITY(actual)=PRIORITY(value) & 0x0F). In addition, there is a bug in the function that AGI uses to determine the order in which to draw objects; normally, objects with lower Y values (meaning toward top of the screen and 'further away') are drawn first. Objects with a fixed priority are assumed by AGI to have an 'effective' Y value based on the priority value, using only the lower nibble - PRIORITY(actual)=PRIORITY(value) & 0x0F).

AGI's algorithm for determining the order based on Y values has a bug, though, so that if objects with fixed priority have a priority value > 22 they won't draw on screen at all.

See the Object Priority topic for a detailed discussion on managing priorities of objects on screen.
