Layout Editor

The layout editor is used to plan the basic room layout of your game. It can be very useful during the design stage of your game, fleshing out the overall flow of the game from room to room. It can also serve as a 'map' of your finished game, showing how all the rooms in your game are connected to one another.

If you don't intend to use the layout editor, you can disable it (on the Layout tab of the Settings window). When disabled, WinAGI will not add comment tags to your logic files, and you will not be able to run the Layout Editor tool.

For help with using the layout editor tools see the Editing Layouts topic.

Creating and Extracting Layouts

WinAGI uses a separate file that contains all metadata needed to support a game's layout. This file is stored in the game's main directory, and uses the game's GameID as the file name with a 'wal' extension.

WinAGI does not automatically create a layout file when a new game is created, unless the new game is created from a template that includes a layout file.

When the Layout Editor is opened and no game layout file exists, WinAGI will extract a layout by analyzing all logics in the game and create the layout file. Once extracted, the layout can then be edited as desired.

Layout Editor Components

For a description of the various components of the Layout Editor window, click on the area of interest:

Toolbar

The toolbar provides access for tools to add and delete rooms and exits, change the IsRoom property to show/hide logics as rooms, and insert transfer points and comment blocks. See the Creating and Editing Layouts for more details on how to use the tools to manage layouts.

Clicking a Tool button once will let you add one object, then the tool will automatically switch back to the Select tool. If you press SHIFT then click the Tool button it will switch to HOLD mode. In HOLD mode, the tool remains selected until you choose another tool, or switch back to normal by clicking the same tool again. The HOLD status is indicated on the status bar.

Layout Drawing Surface

The drawing surface is where the graphical representation of your layout is displayed and manipulated. It includes a grid that helps with positioning items. The grid shows a light dotted dotted line as minor lines, and slightly darker major grid lines. The grid can be toggle on or off by using the Show/Hide Grid command on the Edit menu.

Whether the grid is visible or not, by default objects will 'snap' to the grid when being added or moved.

To change grid settings, including the spacing of grid lines, and the snap feature, use the Layout tab of the Settings window.

Rooms

On the layout editor, each logic that is associated with a 'room' in your game is drawn as a square. A room is typically any area in your game that ego can visit. Rooms usually have a corresponding picture resource.

Arrows represent connections between rooms. Each arrow corresponds to a condition test with a new.room command.

For example, suppose Logic1 has the following code:

if(edgeEgoHit == RIGHT_EDGE)
    {
    new.room(Logic2);
    }

IsRoom Property

Not all logics in a game are used to represent a room. For example, the DeathHandler, Initialization, and GameHelp logics are not associated with places that ego can visit and don't have an associated picture resource. Their IsRoom property should be set to FALSE.

The IsRoom property for a logic can be set via the property window of the Resource Tree. It can also be set using the Show Room and Hide Room commands in the Layout Editor.

Resource ID and Description

When a room is selected, the resource ID and description can be edited by choosing the 'Room Properties' option on the Edit menu.

Picture Thumbnails

By default, if the room has a picture resource with the same number as the room, a thumbnail image of the picture will be included on the Layout display. You can change the default behavior by using the 'Show/Hide All Pics' command on the Resource menu.

Regardless of the default setting, you can show or hide each room's picture thumbnail by using the 'Show/Hide Room Picture' command on the Edit menu.

Use the Edit Picture command on the Edit menu to open the room's picture for editing.

Exits

Arrows between two rooms represent exits. An exit is a condition test (if statement) with a corresponding new.room command.

There are two types of exits. Edge Exits and Other Exits.

Edge exits occur when the condition being tested is reserved variable v2 (ego border code). Edge exits are always drawn from the midpoint of the 'from' room's tested edge to the midpoint of the 'to' room's opposite edge. (For example, if the test is for right edge, the arrow goes from right of 'from' room to the left of 'to' room.)

For any other condition, or if WinAGI is unable to determine what condition is being tested, the exit is known as an Other Exit. Other exits are drawn slightly offset from the midpoint of the rooms' closest edges.

If the 'to' room of an exit is a logic that is not a room (its IsRoom property is FALSE), the exit is drawn as a gray dashed line pointing to nothing. This is a reminder that there is an exit, but that that it points to a 'hidden' logic.

Transfer Points

Often, you will have a layout where rooms 'wrap around', meaning that if the ego character continues moving in one direction, eventually it will 'wrap around' and reach its starting point.

In this case, no matter how you draw your room layout, one exit will have to go all the way across the entire layout.

You may have other situations in your game where exit lines cross over other logics, adding clutter, and making it difficult to understand the layout.

Transfer points allow you to break the exit into two segments. Each segment connects to a uniquely numbered pair transfer points. An exit arrow that 'goes into' one transfer point 'comes out' the other transfer point with the same number.

Error Points

If an exit points to a non-existent or invalid logic, an error marker is used (triangle). You should edit error exits so they point to a valid logic or room.

Comments

Comment blocks (rounded rectangle) can be used to add useful notes to the layout editor.

Integration with Logics

The layout editor and game logics editor are closely integrated. When changes to the layout editor are saved, WinAGI will automatically add or comment out code for any exits that have changed.

When you make and save changes to a logic source file, any changes in exits will be automatically reflected in the layout editor.

Use the Edit Logic command (or double-click on a room or exit) to open the logic for editing. If selecting an exit, the logic editor will set the cursor to the position of the exit in the logic.

Resource List Synchronization

With large layouts, it can sometimes be difficult to find a particular room. You can use the Resource List Synchronization feature to quickly locate any room in your layout. With this feature enabled, selecting a logic in the resource tree with the Layout Editor active will automatically select the corresponding room object in the editor, scrolling the window if necessary. Go to the Layout Settings tab of the game settings to enable or disable this feature.

Comment Tags

The layout editor uses comment tags to keep track of exits in logic source code. These are in the format ##LE001##, ##LE002##, etc. You should not edit these tags. Unless you are deleting the new.room command on the same line, you shouldn't delete them, either.

Repairing Layouts

Despite the built in integration features however, sometimes complex actions, or changes to logics that are made outside of WinAGI can cause the layout editor to get a little out of whack. Use the Repair Layout command on the Resource menu to rebuild and repair the layout.
