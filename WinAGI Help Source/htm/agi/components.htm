AGI Component Files

There are two types of files that make up an AGI game package, the interpreter files and the game resource files. The game resource files, which are platform independent, are described in the Game Resources section.

The interpreter files are platform dependent. The MSDOS version was the most popular and best known interpreter. The platform-specific files for the MSDOS interpreter are:

    • agi: This is the main interpreter file for the MSDOS platform. It contains all the routines for the various AGI commands. In most commercial releases, this file was encrypted with an encryption key that was embedded on a special track on the original game disks, and had to be decrypted by a loader file each time the game was run. When not encrypted, the file was named agi.exe.The size of the file varied from 34Kb in early version 2, growing to 40Kb in version 3.
    • agidata.ovl: In MSDOS, this overlay file contains various bits of data for the AGI such as interpreter error messages and jump tables for the AGI commands inside the AGI file. It is a small file, about 7K bytes (slightly larger in later versions). It is not known for certain why this needed to be a separate file, as it could easily have been combined into a single executable file with the main AGI file. The most likely reason was to allow for some customization between different platforms (but since non-MSDOS versions of AGI have not been analyzed in as much detail as MSDOS versions, this has not been confirmed).
    • sierra.com: In MSDOS, this is the loader which loads (and in many cases would decrypt) the interpreter executable file. When games were installed to a hard drive, the installation batch file would often rename this file with the initials of the game (i.e. kq1.com, sq1.com, etc.) In version 1, this file was called load.
    • *_graf.ovl: One of these five overlay files was loaded when AGI started to provide platform specific graphics related subroutines for each display type (HGC, CGA, EGA, VGA). After detecting the attached video display, AGI would load the appropriate overlay into memory.
    • *_objs.ovl: Beginning with version 2.411, AGI moved the graphics subroutines for managing screen objects to a separate overlay file; one for monochrome (HGC) display, and one for all others.
    • hgc_font: For versions supporting HGC, this file contains character glyph data for displaying characters on screen.

 

Details on interpreter files for other platforms are not included in this help file.

Game Loaders and Copy Protection

The game loader for MSDOS based games was an important part of an AGI game. In most games, the actual interpreter file, agi, was encrypted, as a form of copy protection. The loader file, in conjunction with a specially formatted track 6 on the original Sierra disks that supposedly was not possible to copy using MSDOS disk commands, would decrypt the file, load it into memory, and execute it. An interesting side note is that track 6 was the same for all AGI games. So, when asked to supply an original disk, any disk would work for any game. However, although track 6 was the same for all games, the key value was not. The key offset was then used as a starting point on the special track 6 data, and varied from game to game.

Beginning with the version 3 games, Sierra began to move away from copy protection based on original disks and 'secret' disk data (determined hackers and software pirates eventually found ways to bypass the encryption.) Instead, copy protection based on information found in accompanying game materials became more common. For example, in Gold Rush!, the player had to answer a question with an answer that was found in a paperback book that was included with the game.

When no encryption was needed, the loader file wasn't always present. If no loader file was included, the interpreter executable file was not encrypted, and was run directly to begin the game.

When a loader is executed, it first checks to see if the interpreter executable file is encrypted. It does this by checking the first two bytes of the file. if they are the characters 'M' 'Z' (hex values 0x4D, 0x54 - which is the MSDOS executable file marker), the loader assumes the file is not encrypted. If not, the loader attempts to decrypt the file.

Decrypting the Interpreter File

To decrypt the interpreter executable file, the loader first needed a 128 byte decryption key. The key could either be embedded in the loader file (if the copy protection was removed) or would be read from track 6 of an original Sierra game disk.

The beginning of the loader included a short copyright notice, then the string "keyOfs", then a two-byte key offset code. After that were a string of 128 'k' characters (if the program was either copy-protected, or not encrypted) or 128 bytes of key data (if the key was embedded). A stack space followed (identified by 256 's' characters).

If the key was not embedded (the loader would check for the character 'k' in the first byte of the key space) then the loader would retrieve the encryption key data from the special track 6 of an original game disk.

Once the key was located, the decryption process used is relatively straightforward:

  1. The CARRY bit is zeroed. Set the file pointer to the beginning of encrypted AGI file data. Set key pointer to first byte of the key.
  2. XOR current byte of file data with current key byte.
  3. Rotate current key byte one bit to the right, including the CARRY bit.
  4. Increment key pointer and file data pointer.
  5. Repeat step 2 until 128 bytes are decrypted.
  6. If last rotate resulted in a '1' bit being carried, that bit is ORed into bit 7 of the first key byte.
  7. Reset key pointer to beginning of key. Move file data pointer up 128 bytes.
  8. If the end of the AGI file has not been reached, go back to step 2.

Determining the Encryption Key

For anyone looking to determine the encryption key without an original disk, it actually turns out to be quite trivial due to the characteristics of the resulting decrypted AGI executable. The decrypted AGI file contains a section of all 0x00 values that spans bytes 257 to 384. Since the encryption is a simple XOR function, the corresponding bytes in the encrypted file are exactly* the entire key for the third iteration of decryption. To determine the original key, just rotate the value three bits back to the left. (A utility called AGI Decrypter is available from the creator of WinAGI that uses this method to decrypt any version 2 or 3 MSDOS Sierra game).

 

*If you look closely, you will notice that there is actually a bug in Sierra's decryption algorithm. The CARRY bit is not reset at the start of each loop, and the last bit is only put back into the first byte if the bit is a '1'. This means that at the end of a loop, if the previous loop's last bit (which is now the first byte's most significant bit) was a '1', and the current loop's last bit is a '0', the '1' is not changed back to a '0' - it stays permanently changed to a '1'.

The effect of this is that all the bytes of the key will eventually get all their '0' bits changed to '1' bits, if there are enough iterations of key rotation. This does NOT affect the decryption process since the file was encrypted using the same algorithm, bugs and all. But it does add a tiny complication to breaking the key. After rotating the extracted key data to the left, you need to check the last two bits of the last byte in the key. If those two bits are both '1', then it is possible that the correct value for that byte needs to have bit 1 cleared (i.e. in binary, it should end in 'xxxxxx01', not 'xxxxxx11'). It is easy to determine the correct value of the last key byte, because byte 128 of the DEcrypted file is always a zero. So you can just compare the value of byte 128 in the ENcrypted file to the last byte of the key; if they don't match, just replace the last byte of the key with byte 128 of the ENcrypted file. (AGI Decrypter does this check and always gets the correct key when decrypting a file.)

Game IDs and Interpreter Executable

Each interpreter executable file contained a built-in GameID string that changed from game to game. The set.game.id command, when called, would compare its argument value against the built-in ID. If the IDs did not match, AGI would immediately quit, without prompting or warning. It appears that this was done to prevent players from using the incorrect interpreter file to run AGI games.

In version 3 games, AGI also uses the built-in GameID value as a prefix for the directory file and volume files.

Dealing with Game IDs in Fan-written Games

In order to use existing AGI interpreter files with fan-written games, the GameID issue can be addressed in three different ways:

Method 1: (The hard way.) Use the same technique that Sierra used. Modify the AGI executable file so the built-in GameID matches the ID of the new game. This can be done with a simple hex editor, if you know where to find the built-in ID. Here are a few examples of some game ID's and the data following them:

  • Police Quest: 'P' 'Q' '\x0' 'e' 'I' 'D' 'X'
  • Mother Goose: 'M' 'G' '\x0' 'e' 'I' 'D' 'X'
  • Manhunter 2: 'M' 'H' '2' '\x0' 'I' 'D' 'X'
  • XMAS Demo: 'X' 'M' 'A' 'S' '\x0' 'D' 'X'
  • Leisure Suit Larry: 'L' 'L' 'L' 'L' 'L' '\x0' 'X'

The game ID itself is the null terminated string that ends at the '\x0'. The text that follows it is of no significance, it is simply to fill in the gap although it is useful when searching for the game ID because, as you can see, this text is always "eIDX" (presumably from ``GameIDX'' before being overwritten by the actual ID) or a suffix of it (i.e. "IDX", "DX", or "X"). For most games, the game ID is two or three characters which means that you will be able to rely on the "IDX" string being there for these games.

Because version 3 games require the built in GameID from the interpreter to load the directory and VOL files, fan made games will need to use this approach or they will not run.

Method 2: (An easier way.) The above method will still have the problem where using the wrong interpreter file will crash the game. An easier way to avoid the gameid issue is to never call set.game.id in the first place. Since the command never checks the game's ID against the interpreter executable file's ID, no error is generated, and AGI runs just fine. The downside to this method is that save games will not append the GameID to savegame file names (i.e. they will be 'sg.1', 'sg.2', etc.). This method won't work for fan-written version 3 games.

Method 3: (The easiest way.) But an even easier method is to use a modern AGI emulator, such as AGILE, NAGI or ScummVM. Emulators don't perform the GameID check, and they have added benefits of improved memory management, native support for newer operating systems, and other enhancements. Additionally, most newer interpreters are able to support fan-written version 3 games with no problems.
