OBJECT File

The OBJECT file contains a list of all inventory items that are available to the player during a game. Inventory items usually represent items that a player may pick up and carry during the game. For each item, the file stores the starting room location and the name of the inventory item. See the Inventory Items topic for a more in depth discussion of how inventory items are used within AGI.

A copy of the OBJECT file was included on every floppy disk that was part of a Sierra game. This is because AGI needs the original file whenever a game is restarted.

File Format

The OBJECT file structure consists of

    • a 3 byte file header
    • a table of item entries; each entry is three bytes
    • a series of null-terminated strings, containing the text values of the items

In versions 2.089 and 2.272, the OBJECT file was not encrypted. This meant a curious player could view the file in a hex editor, and possibly see spoilers about what the game might have for items. Beginning in version 2.411, Sierra began using a simple encryption algorithm to hide the file contents from prying eyes.

File Header

The first two bytes of the file header are an offset to the start of text data, measured from byte 3 (i.e. the 3 byte header is ignored). This offset is used by AGI when enumerating through objects so it knows when it has reached the last object.

The third byte of the header is the maximum screen objects value, that determines the maximum number of screen objects that can be loaded at one time during the game.

Item Entries

Following the header was a table of 3-byte values, one for each inventory item in the list. The first two bytes of each entry are an offset to the location of the inventory item's text value. Just like the header offset, all offsets are taken from the start of the item table (not the start of the file). The third byte of each entry is the location (room number) for the item.

Item Text Data

The text data section follows the item entries and is simply a list of NULL terminated strings. The offsets in each table entry point to the first character of each assigned string.

Null Items

The first string in the text section is always a single question mark, that serves as the 'null' item for any items that are considered undefined or unused. The first inventory item (index 0) in most Sierra games is set to the null item (notable exceptions are the two Manhunter games). Other item indexes that aren't used by the game also use the null item. Setting the first item to the null item is not required, but most fan-based games continue the tradition of assigning the null item to item 0. (The null item is always included, even if it is not used by any item in the game.)

Because AGI uses a byte value internally for inventory items, the maximum number of items that can be included in an OBJECT file is 256 (items 0-255).

Item Location

The item location field is usually used to store the room number where an item can be found. It is not required that it be used in this manner. For example, setting an item's location to 1 does not make the item actually 'appear' in room 1. The programmer is still required to write code that makes the item 'appear' in the room.

The special value 255 (0xFF) is used by the item location field to determine which items the player has in inventory (i.e. the item has been 'picked up').

Amiga OBJECT File Format

The OBJECT file in Amiga games had a slightly different format. The header in the file was four bytes, with the same data as in MSDOS but padded with an extra zero byte. The entries in the item data were also four bytes instead of three, again with an extra zero byte added to the end of the entry. The rest of the file followed the exact same format as an MSDOS OBJECT file.
