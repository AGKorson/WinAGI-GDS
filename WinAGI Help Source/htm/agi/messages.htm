Messages

Messages are text data that are used to display prompts and provide textual information to the player. Messages are different from other data types. They are actually a subset of the logic resource data type; they are not stored globally like other variables. Also, messages are read-only; they cannot be modified by AGI as other data types can. Each logic contains a message section at the end of the logic resource. In the message section, the messages are assigned an index value between 1 and 255 (there is no message with index '0').

Messages are represented in source code by the letter 'm' and their index value (i.e. m1, m2, etc.) or by the literal text value of the message. If you attempt to access a message that does not exist (for example, if there are 15 messages in a logic, and you attempt to access message m16) AGI will attempt to access invalid data, which will result in unpredictable output, and will likely crash AGI.

If you attempt to access a message that has not been defined (where the message index is less than the number of messages), AGI will raise trappable error #14. It is not known why this type of invalid message is trapped, while the other type (when index is > message count) does not raise a trappable error.

Generally, each logic's messages are only available to AGI from within its own logic. For example, there is no way for commands in logic 1 to reference the messages in logic 2. The one exception is logic 0; its messages are available to other logics by using the "%g<number>" format code in the calling logic's message, as described below.

Messages generally display the text as it appears within the resource. Format codes and the runtime escape character are used to insert dynamic text and to manage formatting.

Although AGI allows them, carriage returns (ascii code 0x0D) should be avoided. When formatting messages to display in print commands, carriage returns will insert a line break, but they don't reset the count for determining where to insert soft returns, which will cause portions of the message to appear outside the print window. When a hard line break is needed, the new line character (ascii code 0x0A) should be used. (Most AGI development platforms, including WinAGI, use the compiler escape code "\n" to insert a hard line break).

The backspace character (ascii code 0x08) is also allowed in AGI message text, and it will result in deleting the previous character in a string as expected, but isn't of any practical use in a string being displayed on screen.

Message Format Codes

AGI uses the percent sign to insert a format code. The character immediately following the '%' determines the type of format code:

    • %g<number>: the text of the message with this number from message field of logic 0 is inserted at this place.
    • %m<number>: the text of the message with the given number (in this same logic) is inserted at this place.
    • %o<number>: in all versions except 2.089, the name of the inventory item that has an index number equal to the value of the variable given by <number> is inserted at this place. This format code is unlike the others, in that it uses an indirect variable assignment to determine what to print instead of directly referencing the desired inventory item number. As an example, if v99 = 5 and the inventory item with index 5 = "test object" then "%o99" will display "test object" when used in a message. If you entered "%o5" thinking to get "test object" as the output, you would instead get whatever inventory item is referred to by the value in variable v5.
    • %s<number>: the text of the string with the given number is inserted at this place.
    • %v<number>: at this place the output will include a decimal value of variable with the given number. To specify leading zeros, include a vertical bar and the number of desired places. For example, if v34 = 3, then a message with "%v34|2" will display "03" when used in a message.
    • %w<number>: the text of the player-entered word with the given index number is inserted at this place. AGI automatically decrements the <number> value by one, so the index is 'one based'; i.e. first word is %w1, second is %w2, etc.)

The <number> value should be a numerical value, limited to a byte range. If other than a number, AGI will treat the <number> value as zero; i.e. "%mABC" is the same as "%m0ABC", "%vDEF" is the same as "%v0DEF", etc. If the number is greater than 255 it is not converted to a byte value; AGI uses the full two-byte integer value, which will cause overflows when accessing variables, messages, strings, etc. This will cause unpredictable meaningless output.

When using the '|' character to pad variable values, do not use a value greater than 10. AGI uses a fixed array to build the padded string but does not validate the pad value is 10 or less. Pad values greater than 10 will cause AGI to overwrite the buffer, which is likely to crash AGI, or at least display a string of 10 '0' characters instead of the intended variable value.

In most cases, if the character following the '%' character is not one of the above format codes, AGI just skips the percent sign and the character that follows. However, '%d', '%u', '%x' and '%c' are MSDOS library print codes, and if included in a message may occasionally result in unexpected output instead of just skipping.

To display a percent sign, you must precede it with the runtime escape character (using the compiler escape character to generate the runtime escape character):

print("\\%"); [ displays "%"

If you ever need to put a number value right next to a formatted code, use two percent signs to force a break. For example if you want value of v1 to be followed by a '0':

s1 = "%v10"; [ won't work; you'll get the value of v10
s1 = "%v1%%0"; [ this gives desired result

There is also a bug in the text processing code in AGI; if a message ends with a percent sign character, when AGI prepares it for display on the screen, the end-of-message character (ascii code 0x00) gets skipped - this means that the next message immediately following will be added to the output. It is important to be aware of how the percent sign is used by AGI in order to avoid these kinds of unexpected results.

Runtime Escape Character

AGI uses the backslash character '\' as an 'escape' character. When text is processed for display on screen, a backslash is always ignored, and the next character is always displayed, regardless of what it is; even if it is another backslash or a percent sign. No other special feature is provided by the backslash character. AGI does use the backslash character internally when formatting various text elements in the game, but it provides no useful function to game resources, other than to enable display of the backslash and percent sign characters on screen.

Additionally, when saving or restoring, AGI changes the runtime escape character to the '@' symbol, presumably so that when directories are displayed during save and restore operations, the backslashes display correctly. If you use the '@' symbol in a save game description, or a directory location, it will not be displayed (but will still be present as part of the description or directory). When the save or restore operation is complete, AGI changes the runtime escape character back to the backslash.

There are two minor bugs in AGI related to use of the runtime escape character. First, if the WORDS.TOK, OBJECT or a DIR file does not load on the first try, AGI temporarily changes the runtime escape character to '@' in order to display a message that includes backslashes in a directory name to the player. If the player tries again and the file loads on the retry, AGI does not change the runtime escape character back to a backslash. If your game uses the '@' symbol, it won't print correctly during game play in this case.

Second, if a string ends with the runtime escape character '\', and is then subsequently used in a message (with the %s<number> message format code) all characters after the '\' in the string will be replaced with empty spaces. For example, if s1 = "Test \", and m1 = "%s1 more", the output of m1 will actually be "Test      ", not "Test \ more".
