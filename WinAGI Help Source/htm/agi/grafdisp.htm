AGI Graphics Display Mode

In the graphics display mode, AGI displays pictures as a background on which the game is played. Pictures were a graphical representation of the area or 'room' in which the player would interact with the game. Pictures have a resolution of 160 x 168. On most displays, AGI would use a display mode of 320 x 200, and only 16 colors were available. This is apparently a holdover from the IBM PCJr version of AGI, since the PCJr could only support 16 colors in a 160 x 200 resolution. On the newer PC displays, each AGI pixel was actually two screen pixels wide.

The Hercules Graphics Card (HGC) was a notable exception. HGC had a single graphics mode of 720 x 348. Each AGI pixel was 4 x 2 screen pixels. This resulted in black borders 40 pixels wide on each side of the screen. It also meant that on HGC screen, there was only room for a single text row above the graphics picture.

The pictures used as backgrounds for AGI games had a resolution of 160 x 168. This meant that there were 32 rows of pixels not used by the background on EGA and other displays using the 320 x 200 mode. This was done to allow AGI to have space above and below the graphics picture for text-based data, namely the status line, menus, and the player input line. Normally, the backgrounds would be displayed so one 8-pixel high row of text would fit above the display, and three rows below, although this is configurable via the configure.screen command.

On HGC, there were only 12 unused rows of pixels. AGI used these pixels as both the status line and menu line; HGC did not use an input line, and the configure.screen command does not do anything when called in a game using HGC.

AGI did not draw picture resources directly to video display memory. Instead, AGI used a memory buffer to create the images, and would transfer the image to video memory when needed. The image in the buffer actually included both the visual and the priority screen information. Each byte in the buffer held a four bit color value for the visual screen in the lower nibble, and the priority value was stored in the upper nibble. This made it easier for the graphics routines that would manage the placement and movement of screen objects.

Status Line

When enabled, the status line displays the current score and maximum score, and the sound status. The score is displayed on the left edge of the status line, in the format "Score x of y" with x being current score, and y being maximum score. The sound status is displayed on the right, as "Sound is on" or "Sound is off" in version 2.089 and "Sound:on" or "Sound:off" in all other versions. The status line is always displayed as black text on a white background. The colors cannot be changed.

By default, the status line is enabled, and set to row 21. In most Sierra games, the status line is changed to row 0 with the configure.screen command. The status line can be hidden or shown using the status.line.off and status.line.on commands, respectively. After it is displayed, AGI only refreshes the status line when:

    • the status.line.on or status.line.on command is executed;
    • the current score or sound status changes;
    • a new.room, restore.game, toggle.monitor, or graphics command is executed;
    • the graphics screen is redrawn after closing a menu or showing the status screen;
    • or toggling the display type.

The status line is not automatically drawn when the text screen is displayed. Once the text screen is active, AGI will show the status line if it is enabled and the score changes or sound state is toggled.

If the status line is overwritten (for example, by using the display command), AGI does not update it until one of the above conditions occurs.

The current state of the status line (on or off) is not changed during a game restart or a game restore.

Menu Line

When a menu is displayed, it is always on row 0. The row cannot be changed. Unhighlighted menus and menu items are black text on a white background, and highlighted items are inverted. See the Menus Topic for additional information.

Input Line

The input line, where the player can enter text commands, is disabled when AGI starts, and must be enabled using the accept.input command. On non-HGC screens, the location of the input line is row 23 by default, but this can be changed using the configure.screen command. The colors for the input line are set by using the set.text.attribute command.

By default, the entire input line is available for the player to type input. Reserved variable v24 (input length) can be used to limit the input line to less than a full line. See the Input Line discussion in the Keyboard Input topic for more detailed information.

When run on HGC screens, AGI would pause the game, and display an input box, instead of using a dedicated line.

Text on the Graphic Display

In addition to graphics, as described below, text can also be displayed on the graphics screen. The print command is the most common method for displaying text, but will always be black text on a white background with a red border. The display command can be used to show different colors (with some limitations) by using the set.text.attribute command. When using set.text.attribute on the graphics screen, the background color for text is either black (if background color is set to 0) or white (if background color is set to anything else). If the background is white, the foreground will always be black, regardless of the foreground color set by set.text.attribute; if the background is black, then the foreground will be as set.

Due to the way AGI handles passing colors to the DOS functions that draw characters to the screen, it is actually possible to draw black text on non-black backgrounds. If a value > 128 is passed as the foreground color in set.text.attribute, while also passing a value of black as the background color, AGI will use the higher nibble (upper four bits) of the foreground color as the background color, and calls to display will draw black text on colored background. (Extended characters are still garbled - see discussion in the Fonts and Character Codes topic.)

The input line background and foreground colors will be the same as the colors set by the set.text.attribute command as described above. The default values when AGI starts are white foreground and black background.

All text display functions, including internal functions as well as AGI commands such as print and display, use a standard formatting function (presumably a standard C library format function) that uses the '%' character to allow insertion of additional text fields and numbers with a variety of formats. (This is different from the AGI special formatting codes, which only apply to AGI text related commands.) Because of this, you should avoid using any '%'-'char' sequences in any text being displayed other than the supported AGI format codes. See the Message Format Codes topic for additional detail.

Because the graphics screen on HGC does not have as much extra space above/below the background picture, text displayed on the graphics screen will not line up row for row the same as on an EGA display. If you plan to support HGC mode, you should make sure to test any text displays on the graphics screen to be sure it displays on both adapters as you intended.

Character Code Limitation on Graphic Display

As described in the Fonts and Character Codes topic, AGI uses the original IBM character code set to display text characters. However, on the graphics display in most cases, only original 7-bit ASCII characters get drawn correctly.

Priority

All AGI games included the same subtitle, "A Sierra 3-D Animated Adventure". To achieve the effect of 3-D, AGI uses two separate graphics images. The image normally seen on the screen during game play is called the visual image. A second image, called the priority image, exists in memory (and can be shown on screen using the show.pri.screen).

The priority image uses colors to represent the priority of screen elements, so different elements (objects, characters, etc.) would appear to move in front of and behind other elements to give the illusion of 3-D. The priority image also includes control lines to do things like limit the motion of screen objects, trigger events when objects crossed certain lines, and designate areas as water instead of land.

The priority image uses the same sixteen colors that are on the visual image. Four colors (black, blue, green and cyan) are used to represent control lines. The other twelve represent object priority. The higher the color index, the higher the priority. The higher the priority, the closer to the 'front' of the picture the element is determined to be.

Simulating 3-D: 3-D Game on 2-D Display

All screen objects that are added to the screen are assigned their own priority. When AGI draws the screen object, the priority value of each pixel of the background where the object is being drawn is compared to the screen object's priority. If the screen object's priority is the same or higher than the background pixel priority, the pixel is drawn on the visual screen. If it is less, the pixel is not drawn. This is what creates the effect of objects being able to move in front of or behind objects.

The priority screen is also modified in real time to include the screen objects's priority outline. For each pixel that is added to the visual screen, the corresponding pixel on the priority screen is set to the color of the object's priority. The exception to this is that pixels of unconditional, conditional or alarm control lines (color index of 2 or less) are never changed on the priority screen.

The tree has a priority of 11; ego is priority 12, so ego is drawn in front of the tree.

The tree is still priority 11, but now ego is priority 10, so ego is drawn behind the tree.

When two or more screen objects overlap, the screen objects are added in reverse priority order, so the screen object that has the highest priority will be drawn in front of the other. If two screen objects have the same priority, the object with the higher y value (meaning it is 'closer') is drawn last. If the screen objects have the same y value, the screen objects are drawn in index order; the screen object with the higher index number is drawn in front. If a screen object has a fixed priority (by using the set.priority command), it is assigned a 'virtual' y value based on the band that corresponds to its priority value.

Priority Bands

The graphics screen in AGI is divided into bands which are assigned a priority value, with the band at the top given a priority of 4, the next band priority of 5 and so on, with the bottom band given a priority of 14. As screen objects move up and down the screen, AGI automatically updates screen object priority based on the band it is within. This helps create the illusion of 3D.

In version 2.936 and above, the set.pri.base command can be used to adjust the default priority bands.

There are no priority bands 0 - 2 because these values are used for control lines. Priority 3 (water) is the lowest priority, but is not automatically assigned to screen objects based on movement; priority 4 is the lowest priority automatically assigned based on position, and is given to background elements. All objects not 'on water' will be drawn in front of priority 4 elements regardless of their priority.

Band 15 is the highest priority. Unlike the other priority bands, it is not possible for objects to move into this band because movement is limited to 167 in the Y direction. It is possible to assign a priority of 15 to objects using the set.priority command. Band 15 is usually used in pictures for elements that are always in front of any objects on the screen.

Screen objects with a priority of 15 will always ignore all control lines and water/land restrictions regardless of their location. Similarly, if ego is given a priority of 15, it will never set the 'ego on water' or 'ego on signal line' flags.

Avoid using invalid values of priority greater than 15, as it will lead to unexpected results. See the set.priority command for additional details on the negative effects of using invalid priority values.

When drawing background pictures, the artist must pay careful attention to the priority bands, placing elements so they align with the bands so that as screen objects move around the screen, the desired 3-D effect is achieved.

The Horizon

Another feature of AGI that helps create a 3D feel is the horizon. When ego moves to the right or left edge, AGI will set reserved variable v2 (ego edge code) so logic code can use the new.room command to have ego move to other locations in a game while maintaining the effect of moving within a larger world. However, when ego moves up toward the top of the screen, simulating moving further away, the illusion won't work if ego goes all the way to the top. AGI uses a horizon value to make the 3D effect work. The default value for the horizon is 36, which is reset every time the new.room command is called. The horizon can be changed by using the set.horizon command.

Control lines

Control lines are drawn on the priority screen and are used to determine how objects interact with the the background picture. For example, if the background includes a tree, characters shouldn't be able to walk through the tree. A control line drawn around the tree will prevent the character from doing this.

There are three different control lines that provide different interactions: black, blue and green, corresponding to color index values of 0, 1, and 2. (The color cyan [index value of 3] is typically included in control line discussion since it is a special priority value.)

The black (0) control line is an unconditional obstacle. Objects cannot cross over this control line at any time.

The blue (1) control line is a conditional obstacle. Objects won't cross over conditional lines, unless an ignore.blocks command has been issued. The conditional line is useful in situations where some action is required before a screen object is allowed to cross a certain barrier, such as a door. When the door is opened, the ignore.blocks command is issued; now the object can cross the control line and continue moving.

The green (2) control line is an alarm, or signal, line. The alarm line is not an obstacle. It applies only to ego , the player controlled object. When ego touches the alarm line, reserved flag f3 (ego on signal line) is set to TRUE. Code can be written in a logic to respond to the event. For example, if ego touches an alarm line along a cliff, a falling sequence can be initiated.

Water

Cyan (3) is used to represent water (or any area where you wish to confine a screen object). It usually is used to fill a whole area. Water pixels are not actually treated as 'control lines' by AGI. Instead, it is treated the same as other priority levels, one level below 4. This means that if you manually assign a priority of 3 to a screen object (using the set.priority command), the screen object will be 'behind' (hidden) on priority 4 areas, but will be visible on 'water' areas with priority of 3.

AGI provides special handling of 'water' areas, allowing for objects to be restricted only to 'water' areas (using the object.on.water command), or to be prevented from going entirely on to 'water' areas (using the object.on.land).

AGI does not prevent you from using both commands on the same screen object; if you do, AGI will freeze, because the shuffle routine that tries to find a valid position for objects that is called in each interpreter cycle will run in an infinite loop. AGI will also freeze if a screen object is restricted to water on a picture that has no water, or if a screen object is restricted to land on a screen that is entirely water.

Effect of Control Lines on Priority Determination

In AGI, priority areas and control lines are drawn on the same screen. This can lead to problems when AGI needs to know the priority of a pixel that has a control line drawn over it. The way that AGI deals with this is to search downward (increasing Y) until it finds a pixel that isn't a control line and it assumes that the pixel it is looking at is the same priority as the one it found by searching downwards. Usually this will be the pixel immediately below but if a long vertical control line is present, it could be significantly distant. This can cause noticeable visual errors. Careful drawing, and lots of testing is necessary to reduce the effects of control lines on priority.

In some games this can cause a noticeable visual error if you are aware of it. For example, you could walk behind a strip of grass that thinks it is part of the tree below it (KQ1, room 20, blue control line beside left hand tree). There are also a number of other one pixel visual errors that are virtually unnoticeable if you don't know they are there.
