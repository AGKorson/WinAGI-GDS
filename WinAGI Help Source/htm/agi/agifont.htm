AGI Font

AGI used the original IBM character code set (known today as code set 437) for displaying text. The character glyphs that AGI draws when displaying text come directly from DOS (interrupts - which are in different places in memory for the 7-bit characters and the extended characters). This saved Sierra from having to create and manage separate font files. (Except in HGC mode, which managed graphics and text differently, and required a separate font file.)

Regular ASCII Characters

The ASCII character set used in early computers only included 128 characters (based on 7 bits). These characters included normally unprintable control characters (00-31) and all alphanumeric printable characters (32-127). AGI used the built in default DOS font for displaying text:

Extended ASCII Characters

By the time AGI was created, an additional 128 characters were added by using all eight bits. These characters (128-255) were included in DOS on PCs and were referred to as 'extended characters'. The extended characters included letters with diacritics so other languages such as Spanish and French could be represented. There were also a set of characters added that could be used to draw simple boxes and shapes:

The 7-bit characters are the same standard ASCII characters still in use today. The extended characters, however, are different from modern ASCII extended characters. (When AGI was written, there were no Unicode or other multi-byte font character standards.) Sierra games rarely used extended characters though, probably because of the limitations created by the way AGI handles the display of text). (Gold Rush! is one example where extended characters are used to draw lines around text to simulate pages in a book.)

Alternate Code Pages in AGI

The glyphs used for extended characters were determined by the the code page currently being used by MSDOS (originally US/English code page 437). Support for other code pages and keyboard layouts was not added to MSDOS until 1987, so when AGI was originally written, it was not possible to support other languages with keyboard or code page changes. Since the code page only affected the extended character glyphs, and Sierra chose not to support other languages in AGI, the lack of code page support was not a concern. Native support for other language code pages was not addressed by Sierra until they switched to the Sierra Creative Interpreter (SCI).

AGI Character Drawing Function

In early versions of AGI, before menu support was added, Sierra used built in DOS interrupts to display characters for both the text display, and on the graphic display, which meant that all characters, including extended characters, displayed correctly in both modes.

With the introduction of menus, beginning in version 2.411, Sierra needed a way show 'grayed out' text for disabled menu items on the graphics screen. Their solution was to copy the character glyph data from the location where 7-bit ASCII characters are stored, manipulate it to get the effect they wanted and then 'trick' DOS into using the modified glyph data instead of the data from DOS memory.

Sierra also decided to use this method to more easily display black on white text (for example, when using print commands) by inverting the copied glyph data to create a black-on-white image. But because it always copies glyph data from the 7-bit ASCII location, when an extended character is encountered, AGI copies invalid data, resulting in most extended characters appearing as blank spaces, or sometimes a garbled mess.

On the text screen, the DOS character functions remained in use, so extended characters continued to be displayable in text mode.

AGI is also capable of displaying characters for most codes less than 32 (known as control codes). DOS included character glyphs for these characters, and if encountered by AGI, they will display correctly. AGI is capable of displaying a character symbol for all byte codes with the exception of characters 8, 10 and 13. When these characters are drawn on the screen the cursor is moved but no glyph is drawn. Character 8 moves the cursor back one space (erasing the previous character). Character 10 and 13 both add a new line, but when displaying messages, character 13 will not increment the message box height, resulting in text that extends beyond the bottom of the message box.

Text Color Options

The choice of colors available for printing and displaying text in AGI is limited by the underlying DOS functions used to draw characters and the particular DOS display modes that AGI uses when setting up the game. When the text display is active, DOS uses the attributes (set by the set.text.attribute command) as colors; with the upper nibble (higher four bits) as the background color, and the lower nibble (lower four bits) as the foreground color. This is why all possible combinations of foreground and background color are available on the text screen.

On the graphics display, DOS treats the attribute value differently. The lower nibble is still used as a foreground color, but only the highest bit of the upper nibble matters (bit 7). If the bit is cleared, DOS draws the character with a black background using the foreground set by the lower nibble. If the bit is set, then the character is XORed on top of the pixels already present at the character's location. This meant that Sierra had to come up with a work around in order to get black letters on a white background in AGI.

In early versions (2.089 and 2.272), they just drew a white box under the character first, so the XORed characters would correctly show as black on white, including extended characters. However, this only worked when printing messages using the print commands; with the display commands, the underlying graphics pixels are not cleared first, so any background color other than black will just XOR the text pixels on the screen as described above. In these versions, to get text to display as black on white correctly, you need to clear the area (by using clear.lines or clear.text.rect) before displaying the text.

Beginning with version 2.411, they switched to inverting the character's glyph pattern, and then draw it as white on black; so the actual character gets drawn in the background color (black) and the other pixels in the glyph get drawn in the foreground color (white), as described above in the Character Drawing Function discussion. A side effect of the way they did this is that it is not possible to draw text on background colors other than black or white on the graphics display in AGI.
