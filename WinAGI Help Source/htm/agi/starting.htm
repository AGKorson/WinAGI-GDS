Starting AGI

For Sierra's original commercial releases, AGI is started by running a loader file. The loader file uses the original disks to decrypt the main AGI file, which is then run. If the main AGI file is not encrypted, the loader file is not necessary; you can start AGI directly by running the main file AGI.EXE.

Initializing AGI

The first thing AGI does upon startup is to load the main data overlay file, AGIDATA.OVL into memory. Then, command line parameters are processed, if any were included (except version 2.089, which did not include command line options).

Hardware Detection

AGI then does a hardware check to determine the processor and monitor that are present, and number of floppy drives. The hardware check can detect the IBM PC, the IBM PC Junior (PCJr), and the Tandy 1000. Beginning in version 2.903, AGI can also detect if running on the IBM PS/2 with a VGA graphics card.

After determining the type of processor, another check is made to determine which graphics display adapter is present. AGI can detect the Hercules Graphics Card (HGC), IBM Color Graphics Adapter (CGA) and the Enhanced Graphics Adapter (EGA). When the CGA is detected, AGI always begins with the default CGA palette. Beginning in version 2.903, AGI can detect the Video Graphics Adapter card, but only if running on the IBM PS/2. When a VGA card is present on the IBM PC, it will automatically use the EGA compatibility mode of the VGA card.

Memory Allocation

AGI then configures memory by allocating space for the heap. In version 3 only, AGI next allocates 6Kbytes for LZW dictionaries to support decompression. Next AGI allocates space for the picture buffer - 26,880 bytes (160*168). If the display type is HGC, the picture buffer memory allocation is actually twice as big at 53,760 bytes, and a separate font file is loaded. The last bit of memory that AGI allocates is 2560 bytes for the program stack, which is taken from the heap instead of a separate area of memory (because stack variables often need to addressed from within AGI as it processes commands).

System and Interrupt Initialization

AGI then goes through a series of initialization steps, including flushing the keyboard and joystick buffers, loading display-dependent overlay files to support graphics routines. A table is created that will translate screen object vertical position into priority values. AGI then sets the correct video display mode for displaying graphics.

The last step of startup is to install custom MSDOS interrupt vectors to disable the PrintScreen and CTRL+C interrupt functions and handle keyboard input and system timer events.

The system timer interrupt (INT 1Ch) is modified to run exactly 60 times a second, instead of the MSDOS default of 18.2 times per second. AGI then redirects the interrupt to a custom function that controls sound playback (which is why sound units are in 60ths of a second). The custom system timer also calls the 'user' timer (INT 8h) to run once for each three system 'ticks', which is twenty times a second. AGI also installs a custom handler for INT 8h, which is the one that manages the main AGI cycle. This interrupt (custom INT 8h) increments an internal 32 bit counter ('AGI clock ticks') and updates the AGI variables that track clock time (v11, v12, v13, v14). It also checks the stack to see if it has been blown (and will force AGI to quit if it is).

Game Resource Setup

AGI then begins processing the game files. The directory files are loaded on the heap. In version 3 AGI uses the internally stored game ID value to find the game's directory file. If the directory file name doesn't match, AGI will stop loading, display an error message and prompt the user to try again.

After checking for the presence of a joystick, the WORDS.TOK file is then loaded on the heap. All resource linked lists are initialized, then the OBJECT file is loaded.

The screen object table is then created on the heap. At startup, default cycle time and step size for all screen objects is zero; the first new.room or new.room.v command will set the default cycle time and step size to one.

AGI then sets system related variable and flag data - v20 (computer type), v26 (monitor type), f5 (new room), v22 (sound channel count), f11 (noise channel available). Ego is set to player control. Variable v24 (max input length) is set to 41, logic 0 is loaded on the heap and the room.0 pointer is set. The last thing AGI does is to enable sound (sets f9 (sound on) to TRUE), then the interpreter cycle begins.

Command Line Options

It was not widely known, but AGI did include several command line switches that could be used to force some hardware settings. This was mostly unnecessary, though, because AGI could auto-detect the correct, optimal settings. The switches had to be in the form "-x" where x is one of the lower case switch options. Uppercase letters and unrecognized characters, as well as values not preceded by a dash, were ignored. The computer type and the display type were the only two settings that could be set with switches.

Computer Type Switches:

  • -p: This switch forces AGI to PC mode. Obviously, using this on a PC has no effect, since AGI would detect the PC automatically anyway. Using this switch on a non-PC computer will probably result in AGI crashing, but no specific testing has been done to confirm this.
  • -t: This switch forces AGI to Tandy mode. If run on a PC, the graphics will appear the same as a PC with EGA display, but sound will not play, as the hardware for sound is not compatible between a PC and a Tandy.
  • -s: This switch forces AGI to PS/2 mode. If run on a PC, graphics will not display correctly, regardless of the display mode switch chosen. This switch is only valid in version 2.903 and later.
  • Although AGI could run on, and would properly detect the IBM PCJr, Sierra never added a switch to force that mode.

    • Display Type Switches:

  • -h: Forces AGI to use the Hercules Graphics Card (HGC) mode. HGC mode doesn't seem to work with anything other than an actual HGC graphics card (but this has not been thoroughly tested yet). Using this switch when no HGC card (or card with HGC compatibility) is present causes AGI to hang up before it even starts.

    • Although support for HGC was removed in version 3.002.149, the switch was still available. However, using it would cause AGI to crash.

  • -c: This switch forces AGI to use the Color Graphics Adapter (CGA) with the default 'black-cyan-magenta-white' (CGA) palette. This switch can also be used on PCs with EGA and VGA cards, since both those adapters support CGA graphics modes.
  • -r: This switch forces AGI to use the CGA but starts with the alternate CGA palette of 'blue-green-red-brown' (RGB). It can also be used with EGA and VGA display adapters.
  • -e: This switch forces AGI to use the Enhanced Graphics Adapter (EGA). EGA was the most common display adapter during AGI's run. If this switch is used on a computer that doesn't have an EGA or VGA graphics card, AGI will not run correctly.
  • -v: This switch forces AGI to use the Video Graphics Adapter (VGA) card, but ONLY on the IBM PS/2; if used on a PC, AGI will NOT display graphics and text correctly because it will try to use PS/2 specific graphics routines. On a PC, AGI only used the VGA card's EGA compatibility mode. This switch is only valid in version 2.903 and later.
