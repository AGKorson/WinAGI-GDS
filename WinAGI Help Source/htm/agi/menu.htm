Menus in AGI

Menus in AGI

The menu system in AGI is very simple. It includes a single menu bar comprised of one or more menus. The menu bar is always displayed on the topmost line. Menus are added using the set.menu command. Each menu has a list of menu items, which are added using the set.menu.item command. The menu system does not support sub-menus.

Only menu items can be associated with controllers to be able to react to player input; the menus, which are displayed along the horizontal menu bar can never be assigned to controllers.

The five menu related commands (set.menu, set.menu.item, menu.input, enable.item and disable.item ) were added in version 2.272, but did not actually do anything; they all pointed to null functions that just skipped the argument values. In the next version, 2.411, the menu functions were fully implemented and menus became available for the first time.

Creating a Menu System

When creating a menu system in an AGI game, you must follow a specific process. First, use set.menu to add a menu, then add its menu items (using set.menu.item). Repeat for any additional menus. The order that menus and their items are added will determine the order that they are displayed on the menu bar. You cannot move, add or delete menus or menu items once the menu is submitted to AGI.

When each menu or menu item is added, AGI allocates a small memory block (18 bytes for a menu, and 14 bytes for menu items) that is used to keep track of the row/column location, where the menu/menu item is relative to other menu elements, as well as pointers to the message text that will be displayed for the menu or menu item, and the assigned controller number (for menu items only).

Once all menu elements are added, the menu is saved in memory by using the submit.menu command. This leaves the logic with the messages used to set the menu and items by setting the room.0 memory pointer, and secures the menu data on the heap. It is usually most efficient to create and submit menus within logic 0. Also, because the submit.menu command resets the room.0 memory pointer, you should make sure no other resources are currently loaded. Typically menu creation is one of the first tasks that an AGI game does upon starting up.

Do not try to create and submit a menu once AGI has drawn any screen objects on the screen; it will crash AGI. This is because of the way AGI handles the memory allocation for screen object background bitmaps. If a menu is created after objects have been drawn, it will be added AFTER the bitmaps; as soon as AGI updates the screen, the process of rebuilding the background bitmaps will corrupt the memory heap.

The first item of the first menu is the default selection when the menu system is saved.

Do not use the submit.menu command without first assigning menus and menu items. If you do, and then subsequently try to access menus, AGI will crash.

Displaying the Menu System

When the menu system is displayed using the menu.input command, the menu bar is drawn at the top of the screen. Each menu is displayed along the top of the screen, with a single space separating each menu. AGI does not validate that the menus will fit; if the number and size of menus (including spacing) exceeds 40 characters, menus will wrap around to the left of the screen on the second row.

Menus and menu items are always shown as black letters on a white background. The currently selected menu is shown inverted (white on black), and all menu items in the selected menu are drawn underneath the menu text, aligned with the right of the selected menu. If the width of the menu items would go past the right edge, the entire menu is shifted so it aligns right at the edge. The currently selected menu item is also shown inverted.

AGI automatically draws a border around the menu item list when the menu is activated. The width of the menu item list border is determined by the length of the first item. To avoid display anomalies, make sure the first entry is longer than or equal to any other menu item, padding it with spaces, if necessary. Menus should only be used when in graphics display mode. Displaying a menu in text display mode is possible, but menus will not display correctly.

When the menu system is dismissed, the graphics screen underneath it is restored.

Using the Menu System

The menu system is accessed by using the menu.input command (usually by assigning a keyboard shortcut such as the ESC key). When the menu is active, the game cycle is paused. Calling menu.input will only show the menu if all of the following conditions are met:

    • a valid menu has been created, and submitted with the submit.menu command;
    • reserved flag f14 (menu enabled) is set to TRUE;
    • the allow.menu command has not been used to disable the menu (in version 3 games ONLY).

In version 3 interpreters only, the allow.menu command can be used to disable menu input. If the allow.menu command is called with zero as an argument, menus are disabled, regardless of the value of f14. Calling allow.menu command with a non-zero argument will restore normal menu functionality - reserved flag f14 (menu enabled) must still be set to TRUE. One undesirable 'feature' of the allow.menu command is that even though calls to show the menu with the menu.input are ignored when allow.menu has disabled menus, AGI caches the menu display request; once menus are re-enabled by calling allow.menu with a zero value, the menu will be displayed in the next interpreter cycle. There is no way to prevent this, except to be sure not to use the menu.input command when menus are not allowed.

Once activated, the arrow keys (or the joystick, if attached) can be used to navigate among the menus and their items. Pressing the return key will cause the controller for the the selected menu item to be set, and then the menu is dismissed. Pressing the escape key will cancel the menu item selection, and return to normal game play.

Individual menu items can be enabled or disabled as needed during game play (using the enable.item and disable.item commands) in order to present different options based on changing game conditions. It was common in Sierra games to include 'spacer' menu items to separate menu items into groups, and then disable the spacers.

All menu items are enabled by default when created. Restarting a game will re-enable all menus. If you have spacers (or any other menu items you want disabled), they must be re-disabled after a restart.

There is no option to disable and re-enable an entire menu, but any menus that have no items added to them will automatically be disabled by AGI. Normally, disabled menus cannot be selected. They will be grayed out, and AGI will skip over them when the player uses the arrow keys to navigate the menu system. If the first menu is disabled when the menu is saved, it will still be the default selected menu but once the selection is changed, it will no longer be selectable. Disabled menu items are always selectable, but they can never be chosen/activated.
