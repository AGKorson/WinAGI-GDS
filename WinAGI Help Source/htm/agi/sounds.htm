SOUND Resources

Each sound resource contains a five byte (7 byte in the case of version 3 games) header.

The sound specific data follow the header, which varied depending on which platform the resource was intended for. MS DOS and PCJr sounds used a format that matched PCjr hardware. Apple IIgs sounds used customized wave and midi formats.

 

PC/PCJr Based SOUND Resource Format

The format of SOUND resources in the MSDOS version of AGI is based on the sound chip in the IBM PCJr. AGI sounds contain four tracks - three tone tracks and one noise track. Details on how sound is actually generated in AGI games can be found in the AGI Sound System topic.

The sound specific data follow the header, which consists of an eight byte sound channel header, and four tracks of sound data. Each track contains a series of five-byte notes, and each track is terminated by two consecutive 0xFF byte codes. Figure 1 shows how the sound components are arranged in the view resource.

                               Figure 1: Sound Resource Layout

 

The sound header is used by AGI to determine where each track begins. The format is shown below:

Byte  Meaning
0 - 1  Offset to first tone track
2 - 3  Offset to second tone track
4 - 5  Offset to third tone track
6 - 7  Offset to noise track

 

The data in each track are stored as 5-byte notes which contains duration, frequency and attenuation (volume) data:

Byte  Meaning
0 - 1  note duration (little-endian 16 bit word) in 1/60th second increments
2 - 3  frequency data (little-endian 16 bit word)
4  attenuation

 

The first two bytes are the duration as a 16 bit word, in 1/60th second increments. A duration value of 0xffff indicates end of track data.

The third, fourth and fifth bytes of each note are formatted as described in the PCJr sound format. This made it easy to create sound on the PCJr because the bytes could be sent directly to the SN76496A sound generator chip. The two frequency bytes are stored in litle-endian format, meaning the second byte of frequency data actually comes before the first byte in the note data field.

Sierra chose to keep this same format for storing note data on PC/MSDOS systems as well.

Although all four channels of data are present in sound resources that are part of MSDOS games, only the first channel is used, played through the onboard PC speaker. The PC speaker also only had one volume level, so the attenuation values in sound resources (and the global reserved variable v23) are ignored and have no effect.

 

Apple IIgs Based SOUND Resource Format

Apple IIgs sound resources start with the same five byte (7 byte in the case of version 3 games) header. A variable-length IIgs specific header follows, then the sound data. The header is either 54 bytes (for PCM digital sound samples) or 2 bytes (for MIDI sound).

IIgs PCM Digital Sound Sample

Pulse Code Modulation (PCM) is the method used to convert analog sound waveforms into digital signal. The digital data represents the amplitude of the analog wave at intervals defined by the sampling frequency. All Apple IIgs PCM sounds appear to use single channel, 8 bits per sample, with a sampling rate of 8,000 Hz, which provides audio performance similar to old-style analog telephones.

The first two bytes of the PCM header is always the integer value 0x0001. Bytes 8 and 9 contain the size of the data sample. The function of the rest of the data in the header is not known; most of the header appears to be identical for all resources. After the header, the raw PCM data follow, terminated with a single 0x00 byte.

IIgs MIDI Sounds

MIDI sound resources for IIgs are just a raw dump of MIDI data, with a single two-byte header. The value of the header is always 0x0002 to indicate a MIDI sound resource.

The MIDI data appears to be mostly fully compliant with modern MIDI formatting, but there are a few nuances. The resources always begin with program assignments (using message '0xC#') and optionally channel volumes (message '0xB#'). Although the time value of MIDI events is normally passed as a 'delta' value (meaning values greater than 127 get split up into multiple bytes), it appears that IIgs MIDI always used a single byte, even for values greater than 127.

The resources always seem to be terminated with MIDI code '0xFC' (stop sequence). In some cases, the resources include multiple '0xFC' messages, with extra data that follows, but this data appears to be unused data; the sounds always seem to stop at the first '0xFC' message.

In most Sierra IIgs games, there seem to be large number of 'null' resources that are never referenced in logics. Most of these null sounds consist of only four '0xFC' message bytes with no other data.
