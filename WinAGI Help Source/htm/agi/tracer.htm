Trace Mode

AGI includes a command trace mode that can be used to assist in debugging an AGI game. When trace mode is enabled, AGI steps through logics, one command at a time, displaying the command, and the results (TRUE or FALSE) of any test commands. Tracing is enabled by setting reserved flag f10 (tracer enabled) to TRUE. Once enabled, pressing the SCROLL LOCK key toggles the command tracer - press once to begin tracing; press again to close the tracer and process logics in real time.

The trace.on command will also begin tracing. There is no equivalent 'trace.off' command; the only way to exit trace mode is to press the SCROLL LOCK key. Once in trace mode, resetting f10 (tracer enabled) will not exit trace mode either.

While tracing commands, AGI pauses after each command; press any key advance to the next command.

When a return command is reached, the tracer does not display it. It continues with the command following the call command (or in the case of logic 0, it begins a new cycle with the first command in logic 0). AGI actually includes code to display the 'return' command in the tracer window, but due to a bug, that code never gets executed.

You can also press the '+' key (either on numeric keypad, or on the regular keyboard position) to run until the next call command or to the end of the current interpreter cycle, whichever comes first. Tracing then continues with the first command of next logic.

Trace Logic

Before enabling trace mode, the trace.info command should be used to initialize it. This sets the top row and height of the tracer window, and also identifies a logic that AGI will use to display the command names as text values.

AGI uses the messages in the trace logic to display command names. For action commands, the message with the index corresponding to the identified command is used as the command name. For test commands, messages are offset by either 160 (in versions 2.089 and 2.272) or 240; in other words, message 160 (or 240 for higher versions) corresponds to test command 0 and so on.

The trace logic is not automatically loaded when the trace.info command is executed, so you should load it before enabling tracer mode. Load the trace logic from logic 0; if you load it in a logic called by logic 0, the trace logic will get discarded from memory when that logic is discarded. The trace logic also gets automatically loaded whenever a new.room command is encountered or a game is restarted.

It is not necessary to use trace.info, but if it is not used then the command names will not be displayed in the trace window (only the command byte values will be displayed) and the default position and height of the tracer window will be 1 and 15 respectively.

The Trace Window

The trace mode steps through the game's logic commands, one command at a time. It displays a large message box on which the commands that are being executed get displayed.

The trace window is drawn as a large message box. The height and top row are set by passing values to the trace.info command. If the height parameter is less than two, AGI automatically sets it to 2. Since the trace window covers a large part of the screen, it can be useful to use different values for height and top to observe how different parts of the screen are working.

There are three bugs in the trace window code. First, when AGI reaches the point in the interpreter cycle where graphics are updated, any screen objects that are covered by the trace window are drawn on top of the trace window. When the trace window adds a new row, all pixels in the window are shifted up, including the redrawn objects. You may not be able to read some of the previous commands due to this.

Another bug is that the graphic routine that draws the trace window uses the offset of the main graphics window (set with the configure.screen command) twice. The window draws correctly only when the graphics window top is set to row 1; any other value will result in the tracer window being drawn incorrectly.

The line of = characters shows the start of an interpreter cycle. The number to left of the colon is the logic number being traced.

Each line displays the command being executed (using the corresponding message from the trace logic if loaded, or the index number of the command if not) followed by the arguments passed to the command in parentheses. For commands that have any variable arguments, the numeric values of the variables are displayed directly beneath the passed arguments.

Information on the number and type of arguments (number or variable) are stored in a table in the AGIDATA.OVL file.

In this example, the test command if(v29 == 7) shows on the first line as

equaln(29,7)      :F

The actual value of v27 is 0, so below that AGI displays

      (0, 7)      :F

For test commands, An F or T is displayed at the end of a line to show if the test evaluated as 'false' or 'true'. The tracer always displays just the test command result; it will not display a 'NOT' operator. If the test includes 'OR' or 'AND' operators, each test evaluated will display on a separate line in the tracer, with no indication that it was part of an 'OR' or 'AND' operation.

Using the tracing mode can be a good way to get a feel for what happens in each cycle of interpretation.

There is one additional bug with the trace mode. If the trace mode is already active and another trace.on command is encountered, AGI improperly sets the logic command data stream pointer, which usually results in crashing AGI.
