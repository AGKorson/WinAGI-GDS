Overview of AGI

A Short History

AGI (Adventure Game Interpreter) was the Sierra's first large scale game interpreter. Unlike games where the game resources (pictures, sprites (or views), sounds, scripts, etc) were embedded or compiled in the program's executable file, AGI instead used an interpreter that read game resources from a set of resource files. This made it easy for Sierra to create new games, as all they had to do was create a new set of resource files for each new game.

Over the years, AGI was modified and improved. There were three major versions (not many well known games were version 1; WinAGI only supports version 2 and 3 games).

Plaforms

AGI ran on several platforms. Sierra originally created King's Quest I, the first AGI game, for the IBM PCJr. The PCJr ended up as a flop, but Sierra developed AGI to run on other platforms, such as the Tandy 1000, Apple II, Macintosh, Atari, Amiga, and of course, MSDOS on the IBM PC. The huge success of Sierra's games on these other platforms is now a matter of history.

Sierra intentionally dsigned AGI to use (mostly) platform independent game resources (images, scripts, animations, etc.) This allowed them to easily produce games that ran on multiple platforms once they created each platform's executable and support files to read the resources.

WinAGI currently only provides support for creating and editing resources for the MSDOS platform. The reference information in this help file is predominantly aimed at the MSDOS platform implementation of AGI.

AGI Architecture

AGI components typically included the main executable file, a loader file, and supporting files for various graphics displays, sound and other low level support. At the time Sierra created AGI, there wasn't anything else like it. To protect their intellectual property, Sierra employed copy protection and encryption for the main interpreter executable file.

AGI games are sometimes described as 3D graphical adventures, meaning the player could move their character around the screen and in front of and behind objects. Players could use the keyboard and joystick (and on non-MSDOS platforms in later versions, the mouse) to control the main character in the game. This character was know as 'ego'.

The graphics in AGI were low resolution (160 x 200). The screen image that players would see in a game were comprised of two main elements, background pictures, and sprites (or views, as they are known in AGI).

Many players were never aware that AGI actually had four sound channels. On MSDOS based systems, AGI only played one channel through the internal speaker. Other platforms supported all four channels of sound.

AGI works in a loop, or continuous cycle. In each interpreter cycle AGI handles player input (via keyboard, joystick, and/or mouse), assesses current game state by checking various variables and flags, repositions screen objects and redraws the display. Scripts (called logics) are then processed. Logics contain the commands that control the overall game play.

Resources

Along with logics, the other resources included in an AGI game are the background images (called pictures), sprites (animated graphics associated with screen objects called views) and sounds.

Resources are stored in sequentially numbered volume files. Each game has a directory file that AGI uses to locate each resource within the volume files. In version 2 games, a separate directory file for each of the four main resource types is used. Version 3 games combine all directory files into a single file.

AGI games also include WORDS.TOK file that contains a list of the words that a player can use to interact with the game. AGI compares the text that a player types against this list, and acts on them according to commands in logics.

The final resource used by AGI is the OBJECT file. Here 'object' refers to the items that a player can pick up and carry, not the screen objects that are associated with views. These objects are referred to as inventory items to avoid confusion with screen objects.
