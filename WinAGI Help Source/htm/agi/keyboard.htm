Keyboard Input

The primary interface used in AGI is the keyboard. The player uses the keyboard to move ego around the screen, input text commands to the parser, navigate menus/select menu items, and interact with game messages and dialogs.

Whenever a key is pressed, AGI first checks to see if the key is an ego direction key. If it is, the key gets converted to an ego direction input value and then added to the queue. If not an ego direction, AGI adds the key-press to the input queue as a regular key-press.

key-press information is added to the queue as either a two byte keyboard scan code or a two byte ASCII value:

    • Keyboard scan codes are the low-level codes generated by the keyboard when keys are pressed. The first byte of a scan code is always zero. Scan codes are used for non-ASCII key-presses, such as the function keys, ALT key combinations and such.
    • MSDOS will automatically convert key-presses into 2-byte ASCII values when applicable. For ASCII values, the second byte is always zero. ASCII values include control codes (CTRL+A through CTRL+Z), and all basic alphanumeric characters that could be typed on the standard keyboards of the time. See the set.key command topic for a complete discussion on scan codes and ASCII values available in AGI.

At the beginning of each interpreter cycle AGI checks all key-presses in the queue. Any key-presses that match a controller are converted to a controller activation in the input queue. This is why key-presses assigned to controllers never show up on the input line.

Note that this check for controllers only occurs during the check that happens at the beginning of the interpreter cycle. If key-presses are processed outside the normal cycle check (such as while a menu is displayed, or when the have.key or other commands involving keyboard input are called), they are never checked against the controller table.

Any normal key-presses that remain are then added to the input line. The ASCII value of the most recent key-press is then stored in reserved variable v19.

AGI only supports keyboards with US-English layout, and text parsing only responded to the original 127 ASCII characters. Although later versions of MSDOS added the option for other keyboards, Sierra never added support for them to AGI, and AGI will not recognize any non-ASCII characters.

Text Parser

When the player types text commands and presses the ENTER key, AGI parses the text so it can be compared against vocabulary words in the said test command. The process of parsing the text is:

  1. The the player-entered word data arrays are cleared.
  2. AGI removes these special characters from the input string: '`-" (ASCII values 39, 96, 45, 34)
  3. All sequences of more than one space are replaced with a single space. In addition to actual spaces (ASCII value=32), AGI also treats these characters as spaces: ,.?!();:[]{} (ASCII values 44, 46, 63, 33, 40, 41, 59, 58, 91, 93, 123, 125). For example, if a player inputs
    4. Look   at."THE:"[];tree
    the text parser converts it to 
    Look at THE tree
  4. All characters are converted to lowercase as they are loaded by AGI to search for vocabulary words
  5. Starting with the first word of the input, AGI looks up the word in the WORDS.TOK file, trying to find the longest character sequence matching the input line.

    6. If a match is found, AGI checks the word group number:

    • if the word group number is non-zero, the word entered by the player and associated vocabulary word number are stored in the player-entered word data arrays.
    • If the word number is zero (words that are to be ignored, such as 'a', 'an', 'the', etc.) the word is just skipped. Note that the single letter words "a" and "i" are automatically assigned to group 0 (ignored words) if they are not already in the WORDS.TOK file; they can also be assigned to non-zero word group number if desired.

    If the search is unsuccessful (no match found in WORDS.TOK):

    • The unknown word entered by the player is stored in the player-entered word data array.
    • reserved variable v9 is assigned the sequence number of the word in the message that failed to match and the processing ends. Indexing begins at 1, not 0, i.e. first word is '1', not '0'. For example, if the player entered "look tree" and "look" isn't in WORDS.TOK, v9 = 1. If "look" is valid, but "tree" isn't in WORDS.TOK, then v9 = 2.
    • Parsing ends immediately (go to step 7).
  6. If end of input line is reached, or if ten words have been processed, stop parsing. Otherwise, return to step 5 and look for next word.
  7. After parsing ends, if at least one non-zero word is found or an unknown word is encountered, reserved flag f2 (input received) is set to TRUE.

Input Line

The text that is input by the player is displayed on the input line.

 Typical Input Line Showing Input Prompt and Cursor

If running with an HGC display adapter, AGI pauses the game and displays the input line as a box in the middle of the screen. The caption of the box is always "ENTER COMMAND", and cannot be changed. The input line is shown just below the caption, and displays only the text the player enters, and the cursor (if in use). The input prompt is not used in HGC mode.

 HGC Input Display Box

For all other displays, the game continues running while the player inputs commands. The rest of this discussion will focus on non-HGC input.

In versions 2.089 and 2.272, the maximum length of input is fixed at 35 characters, regardless of the length of the input prompt. There is no option to change it. If the input prompt is longer than five characters, the input line will wrap around to the next line if the player enters enough characters.

Beginning in version 2.411, the maximum length of input was changed to be 40 characters, minus the length of the input prompt (and cursor, if one is in use). This usually limits the input to a single line. However, if the input line is on row 24 (the bottom line), a bug in AGI will cause the cursor to jump up one line if the cursor ever reaches the end of the line. The input line will remain on the new row until the ENTER key is pressed. Using the backspace in this situation won't erase the displayed characters on the input line, but will instead just move backward on the row above the input line. Although the text is still visible, it does actually get deleted and won't get parsed.

In versions 2.411 and above, the maximum input length can be shortened by assigning a value to reserved variable v24 (input length). AGI will limit input to the lesser of 40 minus the length of the input prompt (and cursor), or the value of v24. As discussed above, if you want to use row 24 for input, you should set the value for v24 to be 39 minus the length of the input prompt to avoid the bug that makes the input jump up one line.

Input Prompt

When player input is enabled, AGI allows customization of both the input prompt (the character or string displayed at start of the input line) and the cursor (a single character that marks the end of the input line).

The input prompt is the current value of reserved string s0. The input prompt is usually set to a single character ('>' or ']' are most common), but any length string can be used for the prompt. You can also assign a null string to s0 if you do not want any prompt.

The input prompt is off by default when AGI starts. To enable input and display the prompt, you have to use the accept.input command.

If s0 is changed, the input prompt is not immediately updated. AGI only redraws the prompt string after a player presses ENTER, after an accept.input, restart.game, restore.game, save.game, get.number command is executed, or whenever graphics screen is redrawn after using the graphics or status commands, or when a new room is drawn.

Cursor

The cursor is always just a single character, and can be set using the set.cursor.char command. If the message passed as the argument to set.cursor.char is longer than one character, only the first character is used as the cursor. Changes to the cursor will update as soon as any input is made, or the next time the input line is redrawn as described above.

The cursor set by the set.cursor.char command is also used in the save game and restore game dialog boxes.

 

Ego Direction Keys

The nine numbered keys on the numeric keypad are hard coded by AGI to allow the player to control the motion of ego when ego is in player control mode.

Numeric Keypad showing Ego movement keys

When one of these nine keys is pressed, AGI's keyboard handler intercepts it, and adds an ego direction change code to the input queue in place of the key-press. In each interpreter cycle, if AGI processes an ego direction change, the direction value in the screen object table for screen object o0 (ego) is changed to the new value and also copied to reserved variable v6(ego direction).

If ego is stopped, pressing a direction key will set ego's direction to the chosen value. Pressing the same direction key again will stop ego (setting direction value to 0). Beginning with version 2.272, pressing key 5 on the number pad when ego is moving will also stop ego.

If a joystick is attached and is enabled, joystick movement also generates ego movement input.

When a menu is active, ego direction keys (and joystick movement, if a joystick is attached) do not affect ego, but instead are used to navigate the menu system.
