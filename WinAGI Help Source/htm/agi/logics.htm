LOGIC Resources

The heart of Sierra's Adventure Game Interpreter is the LOGIC resource. Logic resources are scripts that contain the code that makes up the game. Each room typically has a picture to represent the graphical image of the room, and a logic script that governs what can take place in that room.

When Sierra created the scripts for its games, they used a text based code editor that used a special AGI script language. An example of a logic script from King's Quest 4 room 7 follows:

if(said( open, door))       [ must be close enough
{
   if(posn( ego, 86, 120, 106, 133))
   {
      if(!night)
      {
         if( door.open)
         {
            print("The door is already open");
         }
         else
         {
            set( game.control);
            set.priority( ego, 11);
            start.update( door);
            end.of.loop( door, door.done);
         }
      }
      else
      {
         print("You can't -- it's locked");
      }
   }
   else
   {
      set( notCloseEnough);
   }
}

 

However, logic scripts are not stored in text format in the game resources. Instead, each command, variable, and program flow element is converted into pseudo code which AGI can understand.

LOGIC Resource Format

Each logic resource contains a five byte (7 byte in the case of version 3 games) header.

The logic pseudo-code data follow the header. The first two bytes of the logic resource are an offset pointer to the beginning of the message text section. Following the offset bytes is the logic byte-code section. At the end of the logic byte-code section is the message text section. Logic 0

The first logic in a game, with index of 0, is unique. This logic is automatically loaded when AGI starts, and is stored permanently in memory as part of the room.0 section on the memory heap. Since it is already loaded, you should not attempt to load logic 0 within your game. Also, do not use the call command on logic 0; it will crash AGI due to recursive looping.

Scan Start

Normally, when a logic is executed, AGI begins processing commands starting at the beginning of the logic. In some cases, you may want to skip over commands on subsequent calls, instead of always starting at the beginning. You can use the set.scan.start command to change the starting location when a logic is executed. When a set.scan.start is encountered, AGI stores the current location in a scan start table; subsequent calls to this logic will begin processing commands immediately after the set.scan.start command. Use the reset.scan.start to restore the scan start back to the beginning of the logic.

Logic Byte-code Section

The logic byte-code section starts immediately after the header and continues until the start of the message text section has been reached. There are three types of codes used in a logic script. There are action command codes, test command codes, and program flow control codes.

Action Command Codes

The first set of codes is the AGI action command set. They range in value from 0 to 181 (0x00 to 0xB5). Although the maximum command value is 181, not all versions used that many commands. See the AGI Versions section for details on the highest command value used for different versions.

Each command value performs a specific action. Some commands have no arguments, others may have as many as 7. When writing source code, there is a corresponding text value associated with each code. For example, "load.pic" corresponds to byte code 0x18 and has one argument; "get.posn" corresponds to byte code 0x27 and has three arguments. Detailed information on each command is available in the AGI Commands section.

If AGI encounters a command byte-code that is higher than the max number of allowed commands, it raises trappable error #16.

Test Command Codes

The second set of codes is the test command set. Test commands range in value from 0 to 18 (0x00 to 0x12) and the number did not change between versions. Just like action commands, some test commands had one or more arguments. Unlike action commands, test commands return a boolean value after AGI evaluates the the arguments. Test commands are used only when AGI has shifted to condition checking mode.

In what appears to be a bug, the number AGI uses to test for invalid test commands is 38, even though there are only 18 valid commands. This means that if an invalid test command value between 18 and 38 is encountered, AGI will crash, instead of raising the trappable error.

Program Flow Control Commands

The third set of commands contain program flow control commands. There are four control commands that range in value from 252 to 255 (0xFC to 0xFF). These codes allow AGI to enter a condition checking mode (so different actions can be taken depending on result of a condition evaluation) and to jump to different positions in the script. The codes translate as

   FF      if 
   FE      else (or goto) 
   FD      not
   FC      or

The if code acts like a set of 'brackets', i.e. the code that makes up the conditional evaluation has an if code at the beginning of the evaluation code and another if code at the end. The following example will illustrate this:

   FF      'if' conditions start
   07      code 07 = 'isset' test command
   05      code 05 = flag 5
   FF      'if' conditions end

The above can be represented in source code as:

if(isset(f5))

which tests whether flag number 5 is set.

When the second if code is encountered AGI switches back to command mode. It then reads the next two bytes. These two bytes contain an offset value. If AGI evaluated the condition in the if statement to be TRUE, AGI ignores the offset, and continues processing commands. If the condition evaluated as FALSE, then AGI will use the offset value to jump over the 'if true' set of commands, as in the following example:

FF 07 05 FF   if(isset(5))
84 00         {                 [for 0x0084 bytes
18 00           load.pic(0);
19 00           draw.pic(0);
1B 00           discard.pic(0);
 ...             ...
              }                 [end of code block; 0x0084 bytes counted.

Multiple if Conditions

More than one test command is allowed in an if statement. When multiple test commands are included in an if statement, they must all evaluate TRUE for the if statement to evaluate TRUE, i.e. the boolean AND operator applies. Use the && operator in source code to show 'AND'ed test commands.

The NOT operator (0xFD) tells AGI to toggle the result of the test command that follows. If the following test command returns TRUE, AGI makes it FALSE, and the if test fails, and vice versa. Use the '!' operator in source code to show 'NOT'ed test commands.

The OR operator (0xFC) tells AGI to evaluate as TRUE if any of the following test commands evaluate as TRUE. The OR operator is used in pairs similar to the if code. This allows AGI to skip past subsequent commands once one is found that returns a TRUE value; AGI reads in bytes until the ending 'OR' code is found.

All these codes can be combined to create complex if statements. Some examples of different combinations of boolean operations on test commands is shown below:

FF 07 55 FF                         if(isset(f85))                               [ one condition tested
FF FD 07 55 FF                      if(!isset(f85))                              [ one condition NOTed 
FF 07 55 07 56 FF                   if(isset(f85) && isset(f86))                 [ multiple conditions, ANDed
FF FC 07 55 07 56 FC FF             if((isset(f85) || isset(f86))                [ multiple conditions ORed
FF FC 07 55 07 56 FC FD 07 57 FF    if((isset(f85) || isset(f86)) && !isset(f87))  [ combination

One major limitation in the way that AGI handles 'OR' and 'AND' operations is that you cannot create tests that group multiple 'ANDed' commands with an 'OR' command, for example

if((isset(f85) && isset(f86)) || isset(f87)) [ INVALID SYNTAX

is not allowed. To create this test using allowed AGI syntax you would have to use multiple if statements:

if(isset(f85) && isset(f86))
   {
   goto DOIT
   }
if(isset(f87))
DOIT:
   {
   [ do stuff 
   }

Else and Goto Statements

Code 0xFE is a bit unusual in that it can act as a goto statement, but is also used in conjunction with the if statement to act as an else statement. The operation that AGI performs when it encounters the 0xFE value is to read the next two bytes as an offset, then jump to that new position. This is why this code is referred to as the goto statement.

When used in conjunction with an if statement, this code acts as an else statement. To create an if ... else construct, the compiler includes a goto statement at the end of the commands to be evaluated if the condition is true. The offset value of this goto statement is the length of commands that are executed if the condition is false (i.e., the else block). An example illustrates this concept best:


FF 07 E7 FF         if(isset(f231))
05 00                 {
65 0F                 print("The door is already open.");
                      }
FE                  else
11 00                 {
0C 24                 set(f36);
77                    prevent.input();
3B 05                 start.update(o5);
03 98 03              assignn(v152, 3);
4C 05 98              cycle.time(o5, v152);
49 05 E8              end.of.loop(o5, f232);
63 46 9A              sound(70, f154);
END:                  }

This is functionally equivalent to:

FF 07 E7 FF        if(isset(f231)) 
05 00                {
65 0F                print("The door is already open.");
FE 11 00             goto END
                     }
0C 24               set(f36);
77                  prevent.input();
3B 05               start.update(o5);
03 98 03            assignn(v152, 3);
4C 05 98            cycle.time(o5, v152);
49 05 E8            end.of.loop(o5, f232);
63 46 9A            sound(70, f154);
:END

Message Text Section

All text that can be printed to the screen from within a logic script is stored in at the end of the logic resource. In version 2 games, and in version 3 games where logics are not compressed, the message text section is encrypted. In version 3 games when the logics are compressed, the message text section is not encrypted.

Messages are referenced by their index position at the end of the logic. The first message is index 1, the second is index 2, etc. The maximum number of messages is 255 (1 - 255).

The format of the text section is as follows:

Byte Meaning
0 Number of messages
1 - 2 Pointer to end of messages
3 - 4
...		 A list of offsets which point to each of the messages.
?? Start of the text data. From this point the messages are encrypted (if version 2); in their unencrypted form, each message is separated by a null character (0x00)

Messages are not required to be assigned in sequential order; you can skip over message numbers. In this case, the message header stores a value of zero for the text pointer. If a logic attempts to access message that is undefined in this manner, AGI will raise trappable error #14.

Note that there is no message with index value of 0; this is because the reference point for the message start offsets is byte 2 (the location following the number of messages byte) of the message section data. The next two bytes, which would be the bytes that a message index of zero would use, instead are used by AGI to mark the end of the message section (so the decryption function knows where to stop). Therefore, zero is not a valid index for messages. If you attempt to reference a message with index of 0, the results will be unpredictable, and may crash AGI.

If you attempt to reference a message with an index value greater than the number of messages in a logic (for example, if the highest message index is 40, and you attempt to use message with index value of 41), AGI will not raise an error. Instead, it returns a pointer value of zero to the function that displays text, which will then use an invalid memory location to try and display the message, resulting in unpredictable behavior, possibly crashing AGI.

See the Messages Data Type topic for information on how to AGI logics interact with messages.
