AGI Scripts

Scripts are used to keep track of resources that are loaded when a game is saved or restored. During the course of a game, as logics, pictures, views and sounds are loaded, script entries are added. Then, when a game is saved, AGI saves the script entries so these resources can be properly re-loaded when the game is restored. Script items also track other picture and view actions, such as unloading, adding to pictures, overlays, etc.

AGI stores script information in a table on the memory heap, which is accessed as needed during save and restore operations.

Script Table

The default script table size is 50 items (100 bytes). This can be changed using the script.size command, but only once, and only if called before any new.room commands are called. Once the table size is set (by calling script.size or after the first new.room call), it cannot be changed.

If you use the script.size command after the script table has already been set the size is not actually changed, but the max script size value that AGI uses to determine when a script overflow error (#11) is raised will be set to the new value. If the new value is larger than the current table size, AGI will allow script entries above the actual limit, which will overwrite memory locations used by other loaded resources, resulting in unpredictable behavior, and most likely crash AGI.

AGI allocates two bytes of memory for each script item in the script table. The first byte indicates the type of script activity, the second indicates the associated resource number. The script table is included in the room.0 portion of the memory heap.

Script Item Types

There are nine resource activities that are tracked in the AGI script:

Script Activity # AGI Command
0 load.logic
1 load.view
2 load.pic
3 load.sound
4 draw.pic
5 add.to.pic*
6 discard.pic
7 discard.view
8 overlay.pic

*add.to.pic actually uses 4 script table entries (total of eight bytes) as follows:

byte 0: script activity number (5)

byte 1: not used

byte 2: view number

byte 3: loop number

byte 4: cel number

byte 5: x position

byte 6: y position

byte 7: priority

Script Management

Each time AGI encounters one of these 9 commands, a script entry is added. It is extremely unlikely to happen, but you should never use one of these commands before the script table is set, as AGI will attempt to store the script entry at an invalid memory location.

AGI maintains a pointer the current script entry. If the number of script entries exceeds the script table size, AGI raises trappable error #11. The script pointer is reset to zero each time a new.room or new.room.v command is called.

During the course of the game, AGI tracks the highest script count, which can be displayed with the show.mem command. This value can be used when debugging to help determine the optimal script size.

Disabling Script Entries

In some cases, it may be desirable to execute commands but not have them add script entries. For example, if a logic repeatedly executes code that loads a view, uses it in some manner, then unloads it, a script overflow could easily be encountered.

Writing to the script table can be disabled by setting reserved flag f7 (script disabled) to TRUE. When this flag is set, no script entries are made to the script table.

Pushing and Popping Scripts

In addition to the block script flag, there are two commands, push.script and pop.script that can be used to manage scripts. These two commands are only available in AGI versions 2.903 and above.

Despite their names, the push.script and pop.script functions are not stack functions. Pushing the script will save the current script table position. If push.script is called multiple times, it will overwrite the stored script position each time it is called. Pushing the script does not affect subsequent additions of commands to the script table.

Popping the script restores the script table position, which effectively deletes all entries int the table past the restored position. After calling pop.script, subsequent script entries will be made starting at the restored position. It is important that push.script be called BEFORE pop.script. Popping the script position without first pushing it could result in unpredictable results since the memory location where the script position is stored may contain an unknown value.

Because AGI relies on the script entries to correctly save and restore games, it is important to exercise caution when manipulating scripts.
