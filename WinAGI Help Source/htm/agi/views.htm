VIEW Resources

VIEW resources contain the graphics information used to display screen objects (sprites) on the graphics screen. Unlike the picture resources which are full-screen background images, view resources are smaller sprites used in the game as animations and objects. They are also stored as bitmaps, whereas pictures are stored in vector format.

Each view resource consists of one or more loops. Each loop in the resource consists of one or more stored bitmaps (cels). Several animations can be stored in one view, or a view can just be used for a single bitmap.

VIEW Resource Format

Each VIEW resource contains a five byte (7 byte in the case of version 3 games) header.

Note: Except when noted otherwise, 16 bit data is stored in little endian format (i.e. the value has the least significant byte stored first, and the most significant byte stored second). Most word values in AGI are stored like this, but not all.

The view data consists of a view specific header followed by loop data for each loop. The view header tells AGI how many loops are in the view and contains offsets to the location of the view description (if there is one) and offsets to the starting position of each loop in the view. Two loop offsets can point to the same loop data. This is done when you want to use mirroring.

Each loop data section starts with a loop header, which tells AGI how many cels are in the loop, followed by cel data. Cel data consists of a cel header, which contains size, transparency and mirroring information. The actual cel data follows the cel header. The view description, if there is one, is located at the end of all cel data. Figure 1 shows how view components are arranged in a view resource.

Details on header information for view components is shown below:

View Header
Byte Meaning
0 Unused (in earlier versions it was the step size for the view)
1 Unused (in earlier versions it was the step time for the view)
2 Number of loops (255 maximum)
3 - 4 Position of description (Both bytes are 0 if there is no description)
5 - 6 Offset to start of first loop data
... Offsets for any other loops follow, up to number of loops given in byte 2

In versions prior to 2.089, step size and step time properties were stored in each view as the first two bytes of the header. When these properties were moved to the screen object structure, the view header was not changed. So the first two bytes of the view header are always ignored in all version 2 and 3 interpreters.

 

Loop Header
Byte Meaning
0 Number of cels in this loop (255 maximum)
1 - 2 Offset to start of first cel data (relative to byte 0 of this loop)
... Offsets for any other cels follow, up to the number of cels given in byte 2

 

Cel Header
Byte Meaning
0 Width of cel in AGI pixels
1 Height of cel in AGI pixels
2 Transparency and mirroring

Loops

Loops are collections of bitmaps (cels) that typically represent frames of an animated screen object's movement; when displayed sequentially, the loop of cels provides the animation that gives life to the screen object. Loops can also be used to store collections of bitmaps for any purpose; AGI provides commands that allow for easy manipulation and display of cels in order to improve the graphical presentation of a game.

The maximum number of loops supported by AGI is 255 (0-254). It is not 256, because AGI uses a byte value to represent the loop count; 255 is the largest integer that can be represented, and since loop numbers are zero-based (first loop is 0, second loop is 1, etc), the 255th loop would have an index of 254, the largest possible index.

Loop Mirroring

Mirroring was introduced beginning in version 2.230 (a very rare version known only to exist in a single demo game). With mirroring, two loops in a view share the same data, but AGI flips (mirrors) the cel image depending on which loop is being displayed. This made it easy to draw animations of character walking left and right -- draw one loop, then let AGI flip it as needed for the opposite direction.

Another advantage of mirroring is saving resource space -- because available resource memory was limited, it had to be used as efficiently as possible.

The original format for mirroring in version 2.230 views used bits 7 and 6 of the cel count byte in a loop's header. AGI would first check bit 7; if bit 7 is not set, then the loop is not a mirror loop.

Bit  Meaning
7 0: Loop does NOT use mirroring
1: Loop uses mirroring
6 0: Track current loop but do NOT flip cels
1: Flip cels when current loop changes
5-4 Loop currently being used
3-2-1-0 Cel count for this loop

Version 2.230 Mirroring Format Using Cel Count Byte

 

If bit 7 is set, bits 5 and 4 are read as a two-bit number (i.e. 0, 1, 2 or 3). This number is compared to the current loop number. If they match, the loop is OK and not mirrored (flipped). If they don't match, the cel count byte is updated to indicate the new loop (i.e. bits 5 and 4 are changed to match the desired loop value). THEN bit 6 is checked; if set, all cels in the loop are flipped; if bit 6 is not set, the cels are not flipped.

The actual cel count in version 2.230 views is always set to the lower nibble of the cel count byte. That means for this version only, views are limited to 16 cels in any loop.

This format for mirroring was only used in version 2.230. Version 2.272, which was used in several commercially released games, moved the mirror control bits into the transparency byte in each cel's header. The reason for the change is unknown, but was likely made to allow loops to have more than 16 cels, and also to reduce clock cycles when the game is running by only flipping cel images when each cel is drawn, rather then doing them all at once everytime a loop changed.

All views in versions 2.272 and beyond used the upper four bits of the transparency byte in the cel header tell AGI whether the cel is in a mirrored loop, and which loop the cel data currently is set to:

Bit  Meaning
7 0: Cel not mirrored
1: Cel is mirrored
6-5-4 Loop currently using this cel
3-2-1-0 Cel transparent pixel color

Version 2.272 and later Mirroring Format Using Cel Transparency Byte

 

Note that the mirror loop number is only three bits wide; meaning only loops 0 through 7 can be mirrored.

When AGI needs to display a cel, it looks at bit 7 to see if its loop is mirrored or not. If it is, then it checks the loop number (bits 6-5-4) -- if this is NOT the same as the current loop, then it flips the cel before displaying it.

To flip a cel, AGI creates a new set of data by flipping the order of the individual data chunks for each row. AGI then overwrites the existing cel data with the new data and updates the cel header mirror/transparency byte with the current loop. When drawn in screen, the cel will now show the flipped version.

If the current loop is changed back to the original loop number, the cell header will tell AGI to flip the cel data back, using the same process.

Because the number of bytes needed to encode a cel can change when it flips, the cel data section in the resource must be large enough to hold whichever image has the larger data size. Figure 3 shows an example making this easier to see:

Figure 3: Mirroring Cel Data

The left-facing image uses 34 bytes to encode the pixel data. But the right-facing image needs 35 bytes. When stored in a resource, if the left-facing cel data is used without an extra byte for padding, when AGI flips it, the right-facing data will overflow one byte into the next cel or loop in the resource. This will almost always cause AGI to crash. Sierra's view editor (and WinAGI) will always determine the size of both images in mirrored cels and make sure the resource is sized correctly to avoid errors.

Autoloop Feature

To facilitate managing loops for screen objects that represent moving characters, AGI has a built-in feature to automatically change the loop based on which direction the object is moving. See the Auto Loop topic for details.

Non-mirroring Loop Sharing

It is allowable for two or more loops to use the same data without being a mirror. In that case, the mirror bit for the cels is NOT set, and both loops will always show the same cels without flipping. There are a few known cases of this in Sierra games. In most cases, there is no value in doing this, but if needed, it is allowed.

 

Cels

The individual bitmaps that make up loops are called cels. A cel can have a maximum width of 160 pixels and maximum height of 168 pixels, the same size as the graphics screen. The maximum number of cels per loop is also 255 (0-254), for the same reason that loops are limited.

Transparency

The least significant four bits of byte 2 in the cel header contain the transparency color index for the cel. This allows irregular shapes to be drawn on the screen even though the cel is rectangular. When the view is drawn on the screen, AGI will not draw any pixels of the cel that match the transparency color, allowing the screen background to show through. All cels have a transparent color, so if you want an opaque cel then you must set the transparent color to one that is not used in the cel.

Cel Data

The actual image data for each cel is stored using run length encoding (RLE) compression. This means that instead of having one byte for each pixel, each byte specifies a chunk of pixels in a row that are the same color. Each time color changes in a row, a new chunk is stored. The end of each row is marked by a byte with value of zero.

Each chunk byte uses the first four bits as the color index, and the last four bits determine the number of pixels in the chunk.

Example: AX BY CZ 00

This line will have:

  • X pixels of color A (AX)
  • Y pixels of color B (BY)
  • Z pixels of color C (CZ)
  • end of the line (00)

 

If the last chunk does not include enough pixels to complete the row, AGI will automatically assign the transparent color to any pixels remaining. For example, if C was the transparent color in the above example, AGI uses AX BY 00. This also helps reduce the size of views in memory.

This method of compression is not very efficient if there are a lot of single pixels in the image (e.g. a view showing static on a TV screen), but in most cases it does save a fair amount of space.

 

View Description

Views are often used in the game to provide a close up image of inventory items that a player might pick up during the game. AGI provides a 'view description' property for views that can be used in conjunction with the show.obj command to display description text in addition to showing the item's image. The description text is stored unencrypted at the end of the cel data, and terminated by a null character. The new line code ('\xA') can be used in the view description if you want hard carriage returns in the text.
