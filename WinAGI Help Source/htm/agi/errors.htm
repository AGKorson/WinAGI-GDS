Internal Error Handling

AGI has a very limited error handling capability. In fact, it is quite easy for a game programmer to crash AGI by creating error situations that go unhandled. It is very important for a game programmer to consider how a player might interact with game elements and to be aware of AGI limitations in order to avoid error situations. There are two main categories of errors that need to be managed, "runtime" errors and "trappable" errors.

Runtime Errors

Runtime errors are those errors that result in unexpected conditions, such as automatic, immediate (and usually unannounced) termination of AGI, freezing of program execution, and reading/writing to invalid memory locations resulting in unpredictable output. There are actually a lot of situations that can cause runtime errors. The most common runtime errors include:

  • Running out of memory: Regardless of the amount of physical memory available, AGI intentionally limited the amount of memory available for running AGI games to approximately 57 kilobytes. Trying to load too many resources or resources that are too large without unloading unnecessary resources, can quickly fill up available memory, and cause an out-of-memory condition. AGI will display a message indicating how much memory it has available as well as how much was requested, and then immediately terminate. See the Memory Management Topic for additional information on how AGI manages memory.
  • Divide by zero: any situation in which AGI does a divide function, where the divisor is zero; for example, if you use the div or div.v with zero divisor. Using incorrect arguments for the random command can also cause a divide-by-zero situation.
  • Loading invalid resources: attempting to load one of the four main resources that does not exist will result in AGI displaying a message warning that the resource does not exist, and then AGI will terminate.
  • Improperly unloading resources: AGI loads resources onto the memory heap sequentially, as a stack. If you unload a resource out of order, the memory stack pointer gets reset to the location of the unloaded resource, and AGI will overwrite later resources. This will result in invalid pointers, and AGI will crash almost immediately. It is critical that resources get unloaded in the order they were loaded.
  • Invalid test command value: If AGI encounters a test command value that is greater than the actual number of valid test commands (18), but less than the value that raises trappable error #15 (38 commands), AGI will attempt to process the test using invalid memory, which will almost always crash AGI. Most AGI logic compilers (including WinAGI) will check for invalid test command values.
  • Invalid parameters to AGI commands: Most AGI commands do not validate that values passed are within a valid range. For example, invalid row/column values for text display commands, invalid screen object numbers for screen object commands, invalid controller numbers, etc. All of these instances will result in unexpected behavior by AGI, and many will crash AGI. See individual command help file entries for specific concerns related to passing invalid parameters.
  • Invalid screen object parameters: There are several ways that incorrect placement and/or configuration of screen objects can cause AGI to lock up. For example, using the object.on.water and object.on.land on the same object or using the object.on.water command for a screen object when the picture has no water pixels. It is important that you are careful in how you place screen objects, and how you configure their movement.
  • Incompatible command on wrong platform: Several AGI commands that were made available in later versions of AGI were not intended to run on the MSDOS platform. The MSDOS interpreter calls a null function when one of these commands (discard.sound, fence.mouse, and mouse.posn) is encountered, but it does not process the command's arguments. AGI will try to process the argument as the next command byte, which will almost certainly crash AGI.
  • Invalid message numbers: Attempting to access an undeclared message with a value that exceeds the message count for a logic results in an null message pointer. This is different than accessing an undeclared message with a value less than the message count, which will instead raise trappable error #14. See the Messages discussion in the Logics Topic for more information.
  • Overflowing the stack: The MSDOS version of AGI has an internal stack (to manage interpreter program flow and DOS based machine code activity) of 2560 bytes. It is not likely, but it is possible to blow the stack if your logics load an excessively high number of resources. Recursively calling a logic is one example of how the stack can be blown.
  • file access errors: The MSDOS version of AGI includes a number of file and disk related error checks, such as write protect, drive not ready, sector not found, read/write errors, etc. When AGI encounters a file access problem (while loading the directory files, WORDS.TOK or OBJECT file; or opening the VOL files, or when extracting resources from the VOL files), AGI displays a message box that shows the error, and gives the player an opportunity to either try again, or to quit AGI. These errors are much less likely when using modern systems running from hard drives and/or other media that is much more reliable than 1980s floppy drives.
  • Invalid loop number: Passing a loop value that is exactly equal to the view's loop count when setting the loop for a screen object. Because the first loop is loop zero, the highest valid value is the loop count minus one. But AGI has a bug in the code that validates the loop value which treats a loop value that is equal to the loop count as a valid loop number. This will almost always crash AGI when it tries to access invalid loop data.
  • Invalid menus: Attempting to display a menu that has not been set up correctly will crash AGI. If submit.menu is called before menus and menu items are set, accessing the menu system by using the menu.input command will attempt to access invalid memory, which crashes AGI. See the Menus in AGI topic for additional details.
  • Invalid string number: AGI does not validate string numbers in commands that can change a string's value. If an invalid string number is used (the limit is 24 [s0 - s23], except for versions 2.089, 2.272 and 3.002.149, which have a limit of 12 [s0 - s11]) AGI will overwrite areas of memory used for other purposes; most of these areas are used internally by AGI to manage all game aspects; if overwritten, your game will almost always crash.
  • Saving a game with too many logics open: When AGI saves a game, it copies the scan information into a table that gets added to the save game file. If more than 30 logics are open when the scan table is created, AGI will overwrite memory that AGI uses for internal memory management, which will most likely crash AGI.
  • Too many items in player inventory: If there are more than 44 items in the player's inventory AGI will attempt to display item text to invalid cursor positions, which will either show garbled data on screen or more likely, crash AGI.

Trappable Errors

There are a few cases where AGI does trap errors, which can then be acted on by the game programmer. When a trappable error occurs, AGI sets the value of reserved variable v17 (error code) to indicate the error number, and provides additional error information in reserved variable v18 (error information). The regular interpreter cycle is interrupted and AGI performs the following steps:

  • any sound resource that is playing is stopped;
  • the heap memory is reset to the room 0 pointer (all loaded resources are dumped). All resource linked lists are also reset;
  • the input queue is cleared;
  • two 'beeps' are sounded on the internal speaker
  • AGI then jumps directly to the point in the interpreter cycle to call logic 0 (it is important for the programmer to recognize that the cycle steps that occur before logic 0 is called are skipped, which may result in unexpected behavior)

Even if an error is trapped, AGI can still crash, depending on whether or not the game programmer deals with the error properly, and resets/reloads needed resources. Because AGI dumps all loaded resources, restarting the game is probably the only viable way to recover from a trapped error.

Almost all Sierra games were written to immediately quit once a trappable error occurred, after displaying the error information to the player. But it is possible to handle the error and continue AGI if the game programmer makes the effort. With clever programming, other recovery options could also be achieved.

Also, note that AGI does not ever reset v17 (error code) or v18 (error information); they are only modified when an trappable error condition occurs. Game programmers must be aware of this when writing code to handle errors, and should reset v17 to zero after handling errors.

 

List of Trappable Errors

AGI version 2.089 (the first version 2 release) on the MSDOS platform included 21 error codes, numbered from 1 to 23, (17 and 22 were not used). Beginning with the next release, version 2.272, three error codes (2, 4 and 7) were dropped, and are were not active in any later versions. The reason why Sierra dropped these three error codes is unknown.

It is also unclear why there are so few trappable errors, and why the particular errors were chosen for trapping. For example, trying to discard an invalid view or picture raises an error (logics and sounds can't be directly discarded by command), but trying to load an invalid resource causes a runtime error and crashes AGI. Invalid parameters for many AGI commands will cause unexpected behavior, yet very few are actually checked and trapped.
